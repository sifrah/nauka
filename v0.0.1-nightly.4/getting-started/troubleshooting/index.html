Troubleshooting

Things break. Networks are unreliable, firewalls are misconfigured, disks fill up. This page covers the most common issues you’ll encounter with Nauka and how to fix them.

First step: always run doctor

Section titled “First step: always run doctor”

Before diving into specific symptoms, run the built-in diagnostic:

Doctor systematically checks every sub-system (WireGuard, control plane, storage, system resources) and reports pass/warn/fail for each. The output usually points directly at the problem, saving you from guessing.

If doctor can’t run (e.g. the state file is corrupted), fall back to checking systemd services manually:

WireGuard interface not up

Section titled “WireGuard interface not up”

Symptom: Doctor reports fabric network as failed. The nauka0 interface doesn’t appear in ip link.

Check the WireGuard config and service:

Common causes:

  • The config file is missing or malformed — this usually means init or join was interrupted. Re-run nauka hypervisor leave (which will clean up partial state) and then init or join again.
  • The WireGuard kernel module isn’t loaded — check with lsmod | grep wireguard. On some older kernels, you may need to sudo modprobe wireguard.
  • Port conflict — another process is using UDP port 51820. Check with sudo ss -ulnp | grep 51820.

If the config exists and looks correct, try restarting:

Peers show as unreachable

Section titled “Peers show as unreachable”

Symptom: nauka hypervisor list shows peers as unreachable. Doctor reports peers as warn or fail, or mesh ping fails.

This means WireGuard handshakes are not completing. The most common causes:

1. Firewall blocking WireGuard traffic

2. Wrong endpoint

If a node is behind NAT or has multiple IPs, the auto-detected endpoint might be wrong. The other nodes will try to connect to the wrong IP. Check what endpoint is configured:

Look at the endpoint field for each peer. If it’s wrong, the fix is to leave and re-init/re-join with --endpoint:

3. Peer’s node is actually down

If WireGuard is stopped, restart it:

4. Asymmetric connectivity

WireGuard requires both sides to be able to reach each other. If node A can reach node B but not vice versa (common with one-sided NAT), the handshake will fail. Both nodes need to have their WireGuard port reachable.

Invalid PIN

Section titled “Invalid PIN”

What it means: The PIN you provided doesn’t match the mesh secret on the target node.

Fix: Get the correct PIN by running peering on the target node — it displays the PIN in the output:

The PIN is the same on every node in the mesh (it’s derived from the mesh secret), so you can run nauka hypervisor peering on any initialized node to see it.

PIN rate-limited

Section titled “PIN rate-limited”

What it means: 5 failed PIN attempts triggered the 10-minute lockout. This is a brute-force protection mechanism.

Fix: Wait 10 minutes and try again with the correct PIN. If you’re not sure of the PIN, check it on the target node with nauka hypervisor peering.

No storage config for region

Section titled “No storage config for region”

What it means: The node is trying to join with --region us, but no S3 storage backend has been published to the control plane for that region. The storage config for a region is set during init — if the mesh was initialized with --region eu, only eu has a storage backend.

Fix: Either join with the region that was configured during init, or configure the additional region’s storage backend.

Control plane issues

Section titled “Control plane issues”

TiKV not starting

Section titled “TiKV not starting”

Symptom: Doctor reports tikv service as failed.

Common causes:

  • Disk full — TiKV stores data in /var/lib/nauka. Check available space:

  • Port conflict — TiKV uses port 20160 on the mesh IPv6 address. Check:

  • WireGuard not running — TiKV binds to the mesh IPv6 address. If WireGuard is down, TiKV can’t start because the address doesn’t exist. Fix WireGuard first, then start TiKV.

PD no leader

Section titled “PD no leader”

Symptom: Doctor reports pd leader as failed. nauka hypervisor status shows pd_members but no leader.

What it means: The PD cluster can’t elect a leader. PD uses Raft consensus, which requires a majority (quorum) of members to be healthy:

PD membersQuorumCan tolerate
110 failures
321 failure
532 failures

If you have 3 PD members and 2 are down, the remaining one can’t form a quorum and won’t elect a leader.

Fix: Bring the failed PD members back online. Check their status:

Re-initialize from scratch

Section titled “Re-initialize from scratch”

If the node’s state is too corrupted to fix, the nuclear option is to leave and start over:

If leave itself fails (e.g. the state file is corrupted and can’t be loaded), you’ll need to clean up manually:

Danger

Manual cleanup deletes all state on this node. Other nodes in the mesh will still have it as a peer — they’ll see it as unreachable until they receive a PeerRemove (which won’t happen, since you’re cleaning up manually) or until they’re told to remove it through some other means. In small meshes, the simplest fix is to have the other nodes run leave and re-create the mesh.
