Nauka is a CLI orchestrator, not a daemon. It configures WireGuard (which runs in the kernel) and systemd services, then exits. There is no `naukad` process to manage separately, no PID file to clean up, and no init script to maintain.

## Directory layout

All runtime and persistent files live under `~/.nauka/`:

```
~/.nauka/                    # mode 0700
├── config.toml              # User configuration (mode 0600)
├── control.sock             # Unix domain socket (when API server runs)
└── data/
    └── state.redb           # Embedded state database
```

The `~/.nauka/` directory is created with mode `0700` (owner-only access) during initialization. This protects the WireGuard private keys and mesh secrets stored within.

## Initialization

The directory is created automatically by `nauka hypervisor init`:

```sh
nauka hypervisor init --name my-mesh --region eu --zone fsn1
```

This creates the `~/.nauka/` directory structure, generates the necessary keys, and sets appropriate file permissions. Running init on an already-initialized node is safe -- it will not overwrite existing keys or configuration.

## Control socket

The control socket is located at `~/.nauka/control.sock`. This is a Unix domain socket used for local communication between the CLI and any running Nauka API server.

If the socket does not exist when a command requires it, Nauka reports that no instance is running and suggests starting one.

## Why this layout matters

As a platform engineer, knowing the directory layout helps you:

- **Back up node state**: the `data/` directory contains the embedded database
- **Troubleshoot permissions**: if Nauka reports permission errors, check that `~/.nauka/` is mode `0700`
- **Inspect configuration**: the config file is always at `~/.nauka/config.toml`
- **Verify initialization**: if `~/.nauka/` exists and contains `config.toml`, the node has been initialized
