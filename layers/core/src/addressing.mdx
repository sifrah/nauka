Every node in a Nauka mesh has an IPv6 address. These addresses are not assigned by a DHCP server or allocated from a central pool -- they are derived deterministically from the node's WireGuard public key. The same key always produces the same address.

## ULA address space

Nauka uses **Unique Local Addresses** (ULA), defined in [RFC 4193](https://www.rfc-editor.org/rfc/rfc4193). ULA addresses begin with `fd00::/8` and are intended for private networks that are not globally routed. They are the IPv6 equivalent of `10.0.0.0/8` or `192.168.0.0/16`.

Each mesh gets a randomly generated `/48` prefix when it is created:

```
fd{XX}:{XXXX}:{XXXX}::/48
```

For example: `fd3a:b7c2:9f01::/48`

This prefix is generated once during `nauka hypervisor init` and stored in the mesh record. With a `/48`, the mesh has 2^80 addresses available -- far more than any realistic deployment will use.

## How node addresses are derived

Each node's address is a `/128` (a single host address) derived from its WireGuard public key:

<Mermaid chart={`

    WG["WireGuard<br/>public key"] --> SHA["SHA-256<br/>hash"]
    SHA --> SUFFIX["Take first<br/>10 bytes"]
    PREFIX["/48 mesh<br/>prefix"] --> ADDR["Node address<br/>/128"]
    SUFFIX --> ADDR

The derivation works as follows:

1. Take the node's WireGuard public key (32 bytes)
2. Compute SHA-256 of the public key
3. Take the first 10 bytes (80 bits) of the hash
4. Combine with the 48-bit mesh prefix to form a full 128-bit IPv6 address

The result looks like this:

```
[48-bit mesh prefix] [80-bit key-derived suffix]
fd3a:b7c2:9f01      :xxxx:xxxx:xxxx:xxxx:xxxx
```

## Why deterministic addressing?

**No address allocation service needed.** A node can compute its own address before it has contacted any other node in the mesh. This matters during bootstrapping -- a node can know its address and configure its WireGuard interface immediately.

**No address conflicts.** Two nodes cannot end up with the same address unless they have the same WireGuard public key, which is cryptographically infeasible. No coordination is needed to assign addresses.

**Auditable.** Given a node's WireGuard public key and the mesh prefix, you can independently verify what address the node should have. There is no address-to-node mapping table that can get out of sync.

## Practical example

A mesh is created with prefix `fd3a:b7c2:9f01::/48`.

Node A joins with its WireGuard public key. The SHA-256 of that key's bytes starts with `e4 7f 2a 01 b3 9c d5 44 a1 f0`. Node A's mesh address becomes:

```
fd3a:b7c2:9f01:e47f:2a01:b39c:d544:a1f0/128
```

Node B joins with a different keypair, producing a different hash suffix. Both addresses live in the same `/48` but at different `/128` endpoints.

You can see the assigned address with:

```sh
nauka hypervisor status
```

## Address stability

A node's address is stable as long as its WireGuard keypair is stable. Keypairs are generated once during `nauka hypervisor init` or `nauka hypervisor join` and do not change under normal operation.

If a node's keypair were rotated (for example, after a suspected key compromise), its address would change. Peers would learn the new address through the peering protocol and update their WireGuard configurations accordingly.

## Prefix utilities

The platform provides several internal operations on mesh prefixes:

- **Extract prefix**: given a full `/128` address, extract the `/48` prefix it belongs to
- **Format prefix**: display a prefix in human-readable form like `fd3a:b7c2:9f01::/48`
- **Check membership**: verify whether a given address belongs to a specific mesh prefix

These are used internally by the hypervisor layer when managing peers and validating addresses during the join process.
