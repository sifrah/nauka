Nauka uses a unified error model across every layer. Whether an error originates from the CLI, the hypervisor, the networking stack, or the storage engine, it always has the same shape and always tells you what to do next.

## Anatomy of an error

Every error includes:

- A **code** -- a stable, machine-readable classification (e.g., `RESOURCE_NOT_FOUND`)
- A **message** -- a human-readable description of what went wrong
- A **suggestion** -- a concrete hint about what action to take
- Optional **context** -- structured key-value pairs with diagnostic detail (resource kind, resource name, etc.)

The suggestion is not optional in practice. Every error that reaches the operator includes guidance on how to resolve it. Silent failures and opaque error codes are not acceptable when managing production infrastructure.

## Error codes

| Code | HTTP Status | Meaning |
|------|-------------|---------|
| `RESOURCE_NOT_FOUND` | 404 | The requested resource does not exist |
| `RESOURCE_ALREADY_EXISTS` | 409 | A resource with that name or ID already exists |
| `VALIDATION_ERROR` | 400 | A field or parameter failed validation |
| `INVALID_NAME` | 400 | A resource name does not follow naming rules |
| `PERMISSION_DENIED` | 403 | The operation is not permitted |
| `CONFLICT` | 409 | The operation conflicts with the current state |
| `PRECONDITION_FAILED` | 412 | A prerequisite was not met |
| `AMBIGUOUS_NAME` | 400 | Multiple resources match the given name |
| `RATE_LIMITED` | 429 | Too many requests -- try again later |
| `INTERNAL_ERROR` | 500 | An unexpected internal error |
| `NOT_IMPLEMENTED` | 501 | The operation is not yet implemented |
| `DAEMON_UNREACHABLE` | 503 | Cannot reach the Nauka process |
| `TIMEOUT` | 504 | The operation did not complete in time |
| `NETWORK_ERROR` | 502 | A network communication failure |
| `STORAGE_ERROR` | 500 | A storage system failure |

## Retryable vs. terminal errors

Errors are classified as retryable or terminal:

- **Retryable**: `TIMEOUT`, `RATE_LIMITED`, `NETWORK_ERROR`, `DAEMON_UNREACHABLE` -- the same request may succeed if retried after a delay
- **Terminal**: `RESOURCE_NOT_FOUND`, `VALIDATION_ERROR`, `PERMISSION_DENIED`, `RESOURCE_ALREADY_EXISTS` -- retrying the same request will not change the outcome

This classification helps automation scripts decide whether to retry or fail immediately.

## What errors look like

In CLI output, errors are printed as human-readable text:

```
Error: hypervisor 'web-node' not found
Suggestion: Start it with: nauka hypervisor init --name <mesh> --region <region> --zone <zone>
```

In JSON mode (for scripting and API responses), the same error is structured:

```json
{
  "code": "RESOURCE_NOT_FOUND",
  "message": "hypervisor 'web-node' not found",
  "suggestion": "Start it with: nauka hypervisor init --name <mesh> --region <region> --zone <zone>",
  "context": [
    ["resource_kind", "hypervisor"],
    ["resource_name", "web-node"]
  ]
}
```

## Ambiguous name resolution

When a name matches multiple resources, the error lists all matches so you can disambiguate:

```
Error: multiple hypervisors named 'web':
  hv-01AAA... (region: eu)
  hv-01BBB... (region: us)
Use the ID directly or add scope flags to disambiguate.
```

## Common errors and their solutions

| Error | Likely cause | What to do |
|-------|-------------|------------|
| `DAEMON_UNREACHABLE` | No Nauka process is running | Run `nauka hypervisor init` or check if the peering server is started |
| `VALIDATION_ERROR` on a duration | Invalid format like `banana` instead of `60s` | Use format: number + unit (`s`, `m`, `h`, `d`) |
| `RESOURCE_NOT_FOUND` for a hypervisor | Node not initialized or wrong name | Run `nauka hypervisor list` to see available nodes |
| `PRECONDITION_FAILED` | A required step was skipped | Read the suggestion -- it will tell you what prerequisite is missing |
