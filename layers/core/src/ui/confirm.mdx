Destructive operations -- deleting a resource, leaving a mesh -- require explicit confirmation before executing. This prevents accidental data loss from a mistyped or misunderstood command.

## Simple confirmation

For standard destructive operations, the CLI displays a warning and asks for a yes/no response. The default is always "No":

```
  ▲  Delete hypervisor hv-west-01? [y/N]
```

Only typing `y` or `Y` proceeds. Pressing Enter or typing anything else cancels the operation.

## Type-to-confirm for critical operations

For operations with severe consequences, the CLI requires the operator to type the exact resource name to confirm. This guards against accidental execution of high-impact commands:

```
  ▲  Delete hypervisor hv-west-01?

  Affected VMs     12
  Affected subnets  3

  This action is permanent and cannot be undone.
  Type "hv-west-01" to confirm: _
```

If the typed input does not exactly match the resource name, the operation is aborted:

```
  Aborted.
```

The impact summary (affected resources, counts) is displayed when available, giving the operator full context before confirming.

## Non-interactive behavior

Confirmation prompts only appear when stdin is a terminal. In non-interactive contexts (piped input, CI/CD), the command should be run with the `--yes` flag to skip confirmation:

```bash
nauka hypervisor leave --yes
```

Without `--yes` in a non-interactive context, the operation will not proceed.

## When confirmation applies

Confirmation is added to operations that are irreversible or have a significant blast radius. Typical examples include:

- Deleting a resource (`nauka hypervisor leave`)
- Leaving a mesh
- Any operation that destroys data or disrupts running workloads

Read-only operations like `list`, `get`, and `status` never prompt for confirmation.
