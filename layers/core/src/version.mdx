Nauka uses semantic versioning with release channel suffixes. The version system supports four release channels, compatibility checking between nodes in a mesh, and binary backup/rollback for safe upgrades.

## Version format

```
{major}.{minor}.{patch}               -- stable release
{major}.{minor}.{patch}-{channel}.{build}  -- pre-release
```

Examples:

```
2.1.0                    (stable release)
2.1.0-rc.1               (release candidate)
2.1.0-beta.14            (beta)
2.1.0-nightly.47         (nightly build)
```

Stable releases have no channel suffix. Pre-release versions include the channel and a build number. A `v` prefix is accepted during parsing but omitted in internal representation.

| Channel | Purpose |
|---------|---------|
| `nightly` | Development builds from the main branch. May have incomplete features. Also recognized as `dev`. |
| `beta` | Feature-complete builds undergoing testing. Suitable for staging environments. |
| `rc` | Release candidate. Code is frozen; only bug fixes are applied. |
| `stable` | Production release. Fully tested and supported. Also recognized as `latest`. |

Channels are ordered by stability: `nightly < beta < rc < stable`. Within the same major.minor.patch, a stable release is always considered newer than its pre-release counterparts.

## Checking the current version

```sh
nauka version
```

The version is determined at build time. CI sets the `NAUKA_VERSION` environment variable during compilation. For local development builds, it falls back to the version in `Cargo.toml`.

## Compatibility between nodes

Before a node joins a mesh or communicates with peers, it checks version compatibility. The rules are straightforward:

- **Major version must match.** A v2.x node cannot join a v3.x mesh. Breaking changes only happen on major version bumps.
- **Minor version differences are allowed.** Nodes with different minor versions can coexist in the same mesh, enabling rolling upgrades.
- **Patch version is always compatible** within the same major.minor.

If there is a minor version difference, Nauka logs a warning but allows the connection. If there is a major version mismatch, the connection is refused.

## Version comparison

Versions are compared in this order:

1. Major version (highest weight)
2. Minor version
3. Patch version
4. Channel stability rank (nightly < beta < rc < stable)
5. Build number within channel

This means `2.1.0` (stable) is always considered newer than `2.1.0-rc.3`, which is newer than `2.1.0-beta.14`, which is newer than `2.1.0-nightly.47`.

## Upgrade safety

Before applying a binary upgrade, Nauka can back up the current binary:

```
~/.nauka/nauka.backup
```

If the new version causes problems, the previous binary can be restored from this backup. The rollback is one level deep -- it restores the single most recent backup, not a full version history.

## Feature flags

Nauka supports runtime feature flags for gating experimental functionality. Feature flags can be enabled or disabled independently of the version, allowing new features to be shipped in a release but activated only when explicitly opted in.

Feature flags default to disabled and must be explicitly enabled. This ensures that experimental behavior never activates on an upgrade unless the operator chooses to turn it on.
