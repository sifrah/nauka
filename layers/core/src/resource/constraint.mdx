Constraints are validation rules that span multiple fields. While each field has its own type validation (a port must be a number, a CIDR must be valid notation), constraints handle the relationships between fields: "if you set this flag, you must also set that one", or "these two options cannot be used together."

Constraints are checked after all individual fields are parsed but before the operation handler runs. If a constraint fails, the operation is rejected with a clear error message explaining what is wrong.

## Constraint types

### Requires

If field A is present, field B must also be present.

```
Error: --private requires --cidr to be specified
```

This is used when one option only makes sense in combination with another. For example, marking a network as private might require specifying a CIDR block.

The `Requires` constraint can also be conditional on a specific value: "if `--type` is `private`, then `--cidr` is required."

### Forbids

If field A is present, field B must NOT be present.

```
Error: --no-auth cannot be used with --token
```

This is a one-directional rule: A forbids B, but B does not forbid A. Use this when one option disables a feature that another option configures.

### Conflicts

Fields A and B cannot both be present. This is bidirectional -- it is an error regardless of which one was specified first.

```
Error: --image and --snapshot cannot be used together. Choose one.
```

Use `Conflicts` for mutually exclusive options. The difference from `Forbids` is that `Conflicts` is symmetric: neither option takes priority.

### OneOf

Exactly one of the listed fields must be present. Not zero, not two -- exactly one.

```
Error: Exactly one of --image, --snapshot, or --blank must be provided.
```

Use this for choices where the user must pick one option from a set.

### AtLeastOne

At least one of the listed fields must be present.

```
Error: At least one of --name, --description, or --tags must be provided.
```

This is useful for update operations where the user must change something but the specific field does not matter.

### Custom

For validation logic that does not fit the patterns above, custom constraints can implement arbitrary checks. For example, validating that a start port is less than an end port, or that a CIDR block does not overlap with an existing allocation.

Custom constraints receive all parsed field values and return either success or an error message.

## When constraints are evaluated

Constraints run at a specific point in the request pipeline:

1. CLI arguments are parsed and type-validated.
2. Interactive prompts fill in any missing required values (when running in a terminal).
3. **Constraints are evaluated.** All constraint violations produce error messages.
4. If all constraints pass, the confirmation prompt is shown (for destructive operations).
5. The handler executes.

This ordering means constraints always see the complete set of field values, including any that were filled in via interactive prompts. It also means that constraint failures never trigger side effects -- the handler has not run yet.

## Constraints are per-operation

Constraints are attached to individual operations, not to the resource as a whole. The same field can be unconstrained in one operation and required in another. For example, `--cidr` might be optional on `create` but required when `--private` is also specified.
