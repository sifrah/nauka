The resource framework is the foundation of Nauka's command-line interface and REST API. Every manageable object in the platform -- a hypervisor, a VPC, a virtual machine -- is declared once as a **resource definition**. From that single declaration, Nauka automatically generates CLI subcommands, API endpoints, input validation, interactive prompts, and formatted output.

## Why a unified framework

Without a resource framework, adding a new manageable object to the platform means writing separate code for:

- CLI argument parsing and help text
- REST API route handlers and request validation
- Output formatting (tables, detail views, JSON)
- Interactive prompts for missing input
- Confirmation dialogs for destructive actions

That quickly leads to inconsistency and duplication. The resource framework eliminates this by deriving all interfaces from one source of truth.

## What a resource definition contains

Each resource definition is composed of several parts:

| Part | Purpose | Learn more |
|------|---------|------------|
| [Identity](/nauka/layers/core/resource/identity) | Name, aliases, and plural form -- determines CLI command names and API paths |  |
| [Scope](/nauka/layers/core/resource/scope) | Where the resource sits in the hierarchy and which parent flags are required |  |
| [Schema](/nauka/layers/core/resource/schema) | Field definitions with types, defaults, environment variable overrides, and visibility |  |
| [Operations](/nauka/layers/core/resource/operation) | Which actions are supported: create, list, get, delete, or custom actions |  |
| [Constraints](/nauka/layers/core/resource/constraint) | Cross-field validation rules checked before any action executes |  |
| [Presentation](/nauka/layers/core/resource/presentation) | Table columns for list output and sections for detail views |  |

## How it works end to end

<Mermaid chart={`
flowchart LR
    A[Resource Definition] --> B[CLI Generation]
    A --> C[API Route Generation]
    B --> D[Argument Parsing]
    C --> D
    D --> E[Constraint Validation]
    E --> F[Handler Execution]
    F --> G[Output Rendering]
`} />

1. At startup, each layer **registers** its resource definitions with a central [registry](/nauka/layers/core/resource/registry).
2. The [CLI generator](/nauka/layers/core/resource/cli-gen) reads the registry and builds the full command tree -- subcommands, flags, help text, and examples.
3. When you run a command, the [dispatch](/nauka/layers/core/resource/dispatch) layer extracts arguments, prompts for missing required values (in interactive mode), validates constraints, handles confirmation prompts, calls the handler, and renders the output.

## What this looks like in practice

Given a resource definition for a hypervisor with `list`, `get`, `init`, `join`, `start`, `stop`, and `leave` operations, the framework automatically generates:

**CLI commands:**

```
nauka hypervisor list
nauka hypervisor get <name>
nauka hypervisor init --mesh-name <name> --endpoint <ip:port>
nauka hypervisor join --target <ip:port> --pin <pin>
nauka hypervisor start
nauka hypervisor stop
nauka hypervisor leave
```

**And equivalent API routes:**

```
GET    /hypervisors
GET    /hypervisors/{name}
POST   /hypervisors/init
POST   /hypervisors/join
POST   /hypervisors/start
POST   /hypervisors/stop
POST   /hypervisors/leave
```

Adding a new operation or field to the resource definition automatically updates both the CLI and the API -- no separate wiring needed.

## Key behaviors for platform engineers

- **Interactive prompts**: When running commands in a terminal and a required argument is missing, Nauka prompts you for it rather than failing. In non-interactive contexts (scripts, CI), missing arguments always produce an error.
- **Confirmation dialogs**: Destructive operations (like `delete` or `leave`) ask for confirmation before proceeding. Use `--yes` or `-y` to skip this in automation.
- **JSON output**: Every command supports `--json` to get machine-readable output instead of formatted tables.
- **Aliases**: Resources can have short aliases (e.g., `hv` for `hypervisor`) for faster typing.
- **Self-registering layers**: The binary itself has no knowledge of specific resources. Each layer registers its own resources at startup. Adding a new resource type requires no changes to the binary.
