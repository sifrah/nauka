Resources in Nauka exist within a hierarchy. A subnet belongs to a VPC. A VPC belongs to a project. Scope defines where a resource sits in this tree and determines which parent identifiers must be provided when operating on it.

## The resource hierarchy

<Mermaid chart={`

    G[Global resources] --> H[hypervisor]
    G --> R[region]
    O[org] --> P[project]
    P --> V[vpc]
    V --> S[subnet]
    V --> VM[vm]
    VM --> VOL[volume]
`} />

Every resource sits at one of these levels. Its scope determines what context is needed to access it.

## Scope levels

### Global scope

Global resources have no parent. They are accessible without any scope flags:

```
nauka hypervisor list
nauka hypervisor get node-1
```

Resources like hypervisors and regions are global because they exist at the platform level, not within a particular organization or project.

### Scoped within a parent

Some resources belong to a parent resource. When a resource is scoped to a parent, the CLI automatically adds the parent flag to every operation:

```
nauka subnet list --vpc production
nauka subnet create my-subnet --vpc production --cidr 10.0.1.0/24
```

In this example, `subnet` is scoped within `vpc`, so the `--vpc` flag is required when creating a subnet and used for filtering when listing.

## How scope affects the CLI

Scope flags are injected automatically into every operation on a scoped resource. You do not need to declare them separately for each operation.

There are two behaviors depending on the operation type:

- **On create**: the parent flag is always required. You must specify which parent the new resource belongs to.
- **On read operations** (get, list): the parent flag may or may not be required, depending on the resource definition. If names are globally unique, the flag can be optional for lookups. If names are only unique within a parent, the flag is required.

For example:

```
# Creating always requires the parent
nauka subnet create my-subnet --vpc production

# Getting may not require it if the name is unambiguous
nauka subnet get my-subnet
```

## Name uniqueness

Scope also controls how name uniqueness is enforced:

| Uniqueness | Meaning | Example |
|------------|---------|---------|
| **Global** | Name must be unique across the entire platform | Hypervisor names |
| **Within parent** | Name must be unique within a single parent | Subnet names within a VPC |

When uniqueness is scoped within a parent, two different VPCs can each have a subnet named `default`. But within a single VPC, subnet names must be unique.

## How scope affects API paths

The scope determines the structure of REST API paths:

**Global resource (no parent):**
```
GET    /hypervisors
GET    /hypervisors/{name}
```

**Resource scoped within a parent:**
```
GET    /vpcs/{vpc}/subnets
POST   /vpcs/{vpc}/subnets
GET    /vpcs/{vpc}/subnets/{name}
DELETE /vpcs/{vpc}/subnets/{name}
```

The parent identifiers are extracted from the URL path and passed to the handler automatically.

## Scope in handlers

When a handler is called, the scope values (parent identifiers) are already extracted from the CLI flags or the API URL. The handler receives them as part of the operation request and does not need to parse them again. If a required scope value is missing or invalid, the request is rejected before the handler runs.
