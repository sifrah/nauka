Dispatch is the execution pipeline that sits between user input and the resource handler. When you run a command like `nauka hypervisor init --mesh-name production`, dispatch takes the parsed arguments, validates them, handles any interactive prompts, calls the handler, and renders the result.

## The dispatch pipeline

<Mermaid chart={`

    A[Parsed CLI arguments] --> B[Extract raw values]
    B --> C{Missing required args?}
    C -->|Yes, interactive terminal| D[Prompt user for values]
    C -->|Yes, non-interactive| E[Return error]
    C -->|No| F[Validate constraints]
    D --> F
    F -->|Fail| G[Return constraint error]
    F -->|Pass| H{Requires confirmation?}
    H -->|Yes, no --yes flag| I["Prompt: Are you sure? [y/N]"]
    H -->|No or --yes| J[Call handler]
    I -->|y| J
    I -->|N| K[Abort]
    J --> L[Render output]
`} />

Each step is described below.

### 1. Extract raw values

The dispatch layer extracts all values from the parsed CLI arguments:

- **Resource name**: the positional argument (for create, get, delete)
- **Scope values**: parent flags like `--vpc`
- **Field values**: all `--flag` options, including schema fields and operation-specific arguments
- **Control flags**: `--json` and `--yes`

### 2. Prompt for missing required values

When running in an interactive terminal (TTY) and a required argument was not provided, Nauka prompts the user instead of immediately failing:

```
$ nauka hypervisor init
? Mesh name: production
? Endpoint: 203.0.113.1:51820
```

This behavior only applies in interactive terminals. In non-interactive contexts (piped input, scripts, CI), missing required arguments always produce an error:

```
$ echo "" | nauka hypervisor init
Error: missing required argument --mesh-name
```

The prompts are type-aware: secret fields mask input, enum fields show the allowed values, and all fields show their description.

### 3. Validate constraints

After all field values are collected (from CLI arguments and interactive prompts), [constraints](/nauka/layers/core/resource/constraint) are evaluated. If any constraint fails, the operation is rejected with an error message.

Constraints are evaluated before the handler runs, so a failed constraint never causes side effects.

### 4. Confirmation prompt

For operations marked as requiring confirmation (typically `delete` and other destructive actions), the user is prompted:

```
Delete hypervisor 'node-1'? This cannot be undone. [y/N]
```

The user must type `y` to proceed. Any other input aborts the operation. The `--yes` / `-y` flag skips this prompt for use in scripts and automation.

### 5. Call handler

The validated request is passed to the handler registered for this resource. The handler receives:

- **Operation name**: which action is being performed (create, list, get, delete, or a custom action name)
- **Resource name**: the target resource name, if applicable
- **Scope values**: parent identifiers
- **Field values**: all parsed and validated field values

The handler performs the actual work (state changes, service management, etc.) and returns a response.

### 6. Render output

The response from the handler is rendered based on the operation's declared output type:

| Response type | Default rendering | With `--json` |
|---------------|-------------------|---------------|
| Single resource | Key-value detail view | JSON object |
| Resource list | Aligned table | JSON array |
| Message | Plain text | `{"message": "..."}` |
| None | Nothing | Nothing |

If the operation defines a success message template, it is rendered with the resource kind and name substituted in:

```
hypervisor 'node-1' created.
```

## CLI and API use the same pipeline

The dispatch pipeline works the same way for both CLI and API requests. The only differences are:

- API requests skip the interactive prompting step (there is no terminal)
- Arguments are extracted from the HTTP request body and URL parameters instead of CLI flags
- Errors are returned as structured JSON instead of terminal-formatted messages

The same handler code runs regardless of whether the request came from the CLI or the API, ensuring consistent behavior across both interfaces.
