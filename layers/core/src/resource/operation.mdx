An operation is an action you can perform on a resource. Nauka provides standard CRUD operations that cover the most common patterns, plus custom actions for anything that does not fit the CRUD model.

## Standard operations

### create

Creates a new resource. The resource name is a required positional argument. All create-only and mutable schema fields appear as CLI flags.

```
nauka hypervisor init --mesh-name production --endpoint 203.0.113.1:51820
```

After a successful create, the CLI displays the new resource's details and a confirmation message.

### list

Lists all resources of this type. Renders output as an aligned table by default, or as JSON with `--json`.

```
$ nauka hypervisor list
NAME     ENDPOINT              STATUS     MESH
node-1   203.0.113.1:51820     active     production
node-2   203.0.113.2:51820     active     production

$ nauka hypervisor list --json
[{"name": "node-1", ...}, {"name": "node-2", ...}]
```

List operations can accept optional filter arguments to narrow results.

### get

Retrieves a single resource by name or ID. Renders as a key-value detail view by default, or as JSON with `--json`.

```
$ nauka hypervisor get node-1
Name:        node-1
Endpoint:    203.0.113.1:51820
Status:      active
Mesh:        production
```

### delete

Removes a resource. Requires confirmation before proceeding:

```
$ nauka hypervisor delete node-1
Delete hypervisor 'node-1'? This cannot be undone. [y/N] y
hypervisor 'node-1' deleted.
```

To skip the confirmation prompt in scripts or automation, use `--yes` or `-y`:

```
nauka hypervisor delete node-1 --yes
```

### update

Modifies an existing resource. Only mutable fields appear as flags, and all are optional -- you only provide the fields you want to change.

## Custom actions

Not everything fits into create/list/get/update/delete. Custom actions handle operations specific to a resource type:

```
nauka hypervisor init --mesh-name production --endpoint 203.0.113.1:51820
nauka hypervisor join --target 203.0.113.1:51820 --pin abc123
nauka hypervisor start
nauka hypervisor stop
nauka hypervisor leave
nauka hypervisor peering
```

Custom actions generate CLI subcommands just like CRUD operations. Each can have its own arguments, constraints, confirmation requirements, and output format.

Examples of custom actions in Nauka:

| Action | Resource | Purpose |
|--------|----------|---------|
| `init` | hypervisor | Create a new mesh on this node |
| `join` | hypervisor | Join an existing mesh via peering |
| `start` | hypervisor | Start the WireGuard service |
| `stop` | hypervisor | Stop the WireGuard service |
| `leave` | hypervisor | Leave the mesh |
| `peering` | hypervisor | Start the peering listener for incoming join requests |
| `status` | hypervisor | Show detailed mesh and connectivity status |

## Output types

Every operation declares what kind of output it produces. This determines how the result is rendered:

| Output type | When used | How it renders |
|-------------|-----------|----------------|
| Resource | Single-item results (get, create) | Key-value detail view |
| ResourceList | Multi-item results (list) | Aligned table |
| Message | Confirmations, status text | Plain text message |
| None | Silent operations | Nothing printed |
| JsonOnly | Machine-oriented output | Always JSON, no table/detail rendering |

The `--json` flag overrides any output type to produce JSON, which is useful for scripting and automation.

## Success messages

Operations can define a success message template that is displayed after the handler completes. Templates support `{kind}` and `{name}` placeholders:

```
hypervisor 'node-1' created.
hypervisor 'node-1' deleted.
```

## Confirmation

Operations can be marked as requiring confirmation. This causes the CLI to prompt before executing:

```
Delete hypervisor 'node-1'? This cannot be undone. [y/N]
```

The `--yes` / `-y` flag is automatically added to any operation that requires confirmation. You do not need to declare it separately.

## Usage examples in help

Operations can include example invocations that appear in the `--help` output:

```
$ nauka hypervisor join --help
Join an existing mesh

Usage: nauka hypervisor join [OPTIONS]

Options:
  --target <addr>  Target hypervisor address
  --pin <pin>      One-time join PIN
  ...

Examples:
  nauka hypervisor join --target 203.0.113.1:51820 --pin abc123
```
