Nauka's CLI is not hand-written. Every subcommand, flag, help string, and example you see in `nauka --help` is generated automatically from the resource definitions registered at startup. This page explains what gets generated and how.

## What gets generated

For each registered resource, the CLI generator produces:

<Mermaid chart={`

    RD[Resource Definition] --> RC[Resource Command: nauka hypervisor]
    RC --> OP1[nauka hypervisor list]
    RC --> OP2[nauka hypervisor get]
    RC --> OP3[nauka hypervisor init]
    RC --> OP4[nauka hypervisor join]
    RC --> OP5[...]
    OP1 --> F1["--json flag"]
    OP2 --> F2["name argument + --json flag"]
    OP3 --> F3["--mesh-name, --endpoint, ..."]
    OP4 --> F4["--target, --pin, ..."]
`} />

1. **A top-level subcommand** named after the resource (e.g., `nauka hypervisor`).
2. **A subcommand for each operation** (e.g., `nauka hypervisor list`, `nauka hypervisor get`, `nauka hypervisor init`).
3. **Flags for each field** on the operation, with types, defaults, and help text.
4. **Automatically injected flags** like `--json`, `--yes`, and scope flags.

## Automatic flags

Several flags are added automatically based on the operation type:

| Flag | Added when | Purpose |
|------|-----------|---------|
| `--json` | On `get` and `list` operations | Output raw JSON instead of formatted tables |
| `--yes` / `-y` | On operations marked as requiring confirmation | Skip the interactive confirmation prompt |
| Scope flags (e.g., `--vpc`) | On all operations for scoped resources | Identify the parent resource |

These flags do not need to be declared in the resource definition -- they appear automatically.

## How operation types affect the CLI

Each operation type generates a different argument structure:

### create

- A required positional `name` argument (the resource name, 3-63 characters, lowercase alphanumeric and hyphens)
- All create-only and mutable schema fields as `--flag` options
- Scope flags if the resource has a parent

### get

- A required positional `name` argument (resource name or ID)
- The `--json` flag
- Scope flags if the resource has a parent

### list

- The `--json` flag
- Scope flags if the resource has a parent
- Any filter arguments defined on the operation

### delete

- A required positional `name` argument
- The `--yes` / `-y` flag
- Scope flags if the resource has a parent

### update

- A required positional `name` argument
- All mutable schema fields as optional `--flag` options
- Scope flags if the resource has a parent

### Custom actions

- Whatever arguments are defined on the operation (required or optional)

## Aliases

Resource aliases are registered as command aliases. If `hypervisor` has the alias `hv`, both of these work:

```
nauka hypervisor list
nauka hv list
```

## Help text and examples

Help text is derived from field descriptions and operation descriptions:

```
$ nauka hypervisor init --help
Create a new mesh on this node

Usage: nauka hypervisor init [OPTIONS]

Options:
  --mesh-name <name>    Name for the new mesh
  --endpoint <addr>     Public endpoint for this node
  --listen-port <port>  WireGuard listen port [default: 51820]
  -h, --help            Print help

Examples:
  nauka hypervisor init --mesh-name production --endpoint 203.0.113.1:51820
```

The "Examples:" section comes from examples defined on the operation.

## Output rendering

The CLI generator also handles rendering responses from handlers:

- **Table rendering** for list operations: columns are sized automatically based on content, headers are printed with a separator line, and empty results show a configurable message.
- **Detail rendering** for get operations: fields are displayed in labeled sections with consistent alignment.
- **Value formatting**: bytes are shown as KiB/MiB/GiB, timestamps as UTC dates, durations as human-readable strings, booleans as yes/no, and sensitive values are masked.

All rendering is bypassed when `--json` is used -- the raw JSON from the handler is printed directly.

## Self-assembling CLI

The binary itself has no knowledge of which resources exist. At startup, it collects all registered resource definitions from the registry and generates the command tree. Adding a new resource to a layer automatically makes it available as a CLI command with no changes to the binary.
