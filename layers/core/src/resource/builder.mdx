Resource definitions are the single source of truth for everything Nauka knows about a resource type. Each definition is assembled by composing several parts: identity, scope, fields, operations, and presentation.

## Anatomy of a resource definition

<Mermaid chart={`
flowchart TD
    RD[Resource Definition]
    RD --> ID[Identity: name, aliases, plural]
    RD --> SC[Scope: parent hierarchy, uniqueness]
    RD --> SF[Schema Fields: typed input fields]
    RD --> OP[Operations: create, list, get, delete, custom]
    RD --> PR[Presentation: table columns, detail sections]
`} />

When building a resource definition, you specify each of these parts:

### 1. Identity

The resource's name, CLI command name, aliases, and plural form. See [Identity](/nauka/layers/core/resource/identity) for details.

- **Name**: the internal identifier (e.g., `hypervisor`)
- **CLI name**: what users type (usually the same as the name)
- **Plural**: for API paths and messages (auto-generated as name + "s" if not set)
- **Aliases**: short alternatives (e.g., `hv` for `hypervisor`)

### 2. Scope

Where the resource sits in the hierarchy. See [Scope](/nauka/layers/core/resource/scope).

- **Global**: no parent required (e.g., hypervisors)
- **Within a parent**: requires a parent flag on CLI operations (e.g., subnets within a VPC)

Parent references define the CLI flag name, whether the parent is required on create, and whether it is required on read operations.

### 3. Schema fields

The fields a resource has -- each becomes a CLI flag. See [Schema](/nauka/layers/core/resource/schema).

Fields define their type, description, mutability, optional short flag, default value, environment variable override, and help visibility.

### 4. Operations

What you can do with the resource. See [Operations](/nauka/layers/core/resource/operation).

Standard CRUD operations (create, list, get, delete) are available as presets. Custom actions (init, join, start, stop, leave, etc.) can be added for resource-specific behavior. Each operation can have its own arguments, constraints, confirmation requirements, examples, and output format.

The shorthand `crud` adds all four standard operations (create, list, get, delete) at once.

### 5. Presentation

How the resource appears in CLI output. See [Presentation](/nauka/layers/core/resource/presentation).

- **Table columns**: which fields appear in `list` output, their alignment, width, and display format
- **Detail sections**: how `get` output is organized into labeled sections with formatted fields
- **Empty message**: what to show when a list result is empty
- **Default sort**: which field to sort table results by

## How definitions are used

Once a resource definition is built, it is registered with the [resource registry](/nauka/layers/core/resource/registry) alongside a handler function. From that point:

1. The [CLI generator](/nauka/layers/core/resource/cli-gen) reads the definition to produce the full command tree with all flags, help text, and examples.
2. The [dispatch layer](/nauka/layers/core/resource/dispatch) uses the definition to parse arguments, validate constraints, manage prompts and confirmations, call the handler, and render the output.
3. The API route generator uses the definition to create REST endpoints with the correct paths and parameters.

Adding a new field, operation, or presentation column to a resource definition automatically updates all interfaces -- no separate wiring required.
