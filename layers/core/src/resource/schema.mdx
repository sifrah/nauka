The schema defines what fields a resource has. Each field becomes a CLI flag (like `--endpoint` or `--cidr`) and a corresponding API parameter. The framework handles parsing, default values, environment variable overrides, and help text generation automatically.

## Field types

Every field has a type that determines how user input is parsed and validated:

| Type | CLI appearance | Description |
|------|---------------|-------------|
| String | `--name <value>` | Free-form text |
| Integer | `--port <number>` | Whole number |
| CIDR | `--cidr <block>` | Network CIDR notation (e.g., `10.0.0.0/16`) |
| Port | `--port <number>` | Network port number |
| SizeGb | `--size <gb>` | Disk or storage size in gigabytes |
| SizeMb | `--memory <mb>` | Memory size in megabytes |
| Duration | `--timeout <duration>` | Time duration |
| IpAddr | `--endpoint <ip>` | IP address |
| Path | `--config <path>` | File system path |
| Flag | `--force` | Boolean toggle (present = true, absent = false) |
| Enum | `--type <value>` | One of a fixed set of allowed values |
| KeyValue | `--label key=value` | Key-value pair, can be specified multiple times |
| Secret | `--password <value>` | Sensitive value, masked in interactive prompts |
| ResourceRef | `--vpc <name>` | Reference to another resource, resolved by name or ID |

## Defaults

Fields can have default values. When a field has a default, it is effectively optional -- the default is used if the user does not provide a value. Defaults are displayed in the `--help` output:

```
$ nauka hypervisor init --help
...
Options:
  --listen-port <port>  WireGuard listen port [default: 51820]
  --mesh-name <name>    Name for the new mesh
```

## Environment variable overrides

Fields can be configured to accept their value from an environment variable. This is useful for values that should not appear in command history or that differ between deployment environments:

```
$ export NAUKA_ENDPOINT=203.0.113.1:51820
$ nauka hypervisor init --mesh-name production
# The --endpoint flag is populated from NAUKA_ENDPOINT
```

When both a CLI flag and an environment variable are provided, the CLI flag takes precedence.

## Short flags

Frequently used fields can have single-character short flags for convenience:

```
nauka hypervisor init -n production    # same as --mesh-name production
```

## Field mutability

Each field has a mutability setting that controls when it can be set:

| Mutability | Meaning |
|------------|---------|
| **Create-only** | Can be set at creation time but not changed afterward (e.g., CIDR block) |
| **Mutable** | Can be updated after creation (e.g., description, labels) |
| **Read-only** | Computed by the system, never set by the user (e.g., ID, creation timestamp) |
| **Internal** | Used internally, never exposed in the CLI or API |

Create-only and mutable fields appear as CLI flags on the `create` command. Only mutable fields appear on the `update` command. Read-only and internal fields are never shown as input flags.

## Help visibility

Fields can be configured with different visibility levels in CLI help output:

| Visibility | Behavior |
|------------|----------|
| **Normal** | Shown in both `-h` (short help) and `--help` (full help) |
| **Advanced** | Only shown in `--help` (full help), hidden from `-h` |
| **Hidden** | Never shown in help output, but still accepted as a flag |

This keeps the short help clean for common usage while making advanced options discoverable with `--help`.

## Enum fields

Enum fields restrict input to a predefined set of allowed values. The CLI shows these as possible values in help output and rejects any value that is not in the set:

```
$ nauka vm create web-1 --type standard
$ nauka vm create web-1 --type invalid-type
error: invalid value 'invalid-type' for '--type <type>'
  [possible values: standard, performance, memory-optimized]
```

Enum fields can also have a default value that is used when the flag is not provided.
