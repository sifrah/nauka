Nauka's cryptographic primitives handle three concerns: generating WireGuard keypairs for tunnel encryption, managing mesh secrets for trust bootstrapping, and deriving purpose-specific keys from shared secrets.

## WireGuard keypairs

Every node in the mesh has a WireGuard keypair based on Curve25519. The keypair is generated during `nauka hypervisor init` (for the first node) or `nauka hypervisor join` (for subsequent nodes).

- The **private key** stays on the node and is never transmitted
- The **public key** is shared with peers so they can establish encrypted tunnels

Keys are encoded as Base64 strings, which is the standard format that WireGuard uses. You can see a node's public key in the output of:

```sh
nauka hypervisor status
```

You do not need to manage these keys manually. Nauka generates them, stores the private key locally, and distributes the public key to peers automatically through the peering protocol.

## Mesh secrets

A mesh secret is a 32-byte shared secret that bootstraps trust between nodes. It is the single credential needed to join a mesh.

Mesh secrets are displayed with a `syf_sk_` prefix followed by a Base58-encoded value:

```
syf_sk_7Kz3x9vPq2mN5wR8tY4uJ6hB1fD0gC3eA8sL2nM4pQ6r
```

The `syf_sk_` prefix makes mesh secrets visually distinct from other credentials and allows automated scanning tools (such as secret detectors in CI pipelines) to detect accidental exposure in logs, environment variables, or source code.

A mesh secret is generated once when the first node creates the mesh with `nauka hypervisor init`. The operator then provides this secret to other nodes when they join with `nauka hypervisor join`.

:::caution
The mesh secret is the root of trust for the entire mesh. Anyone who has it can join a new node. Store it securely and rotate it if you suspect it has been compromised.
:::

## Key derivation

The mesh secret is never used directly for encryption. Instead, Nauka derives purpose-specific keys from it using HKDF-SHA256 (HMAC-based Key Derivation Function). Each derived key is bound to a specific purpose through a domain string:

<Mermaid chart={`
flowchart TD
    SECRET["Mesh Secret<br/>(32 bytes)"] --> HKDF["HKDF-SHA256"]
    HKDF -->|domain: peering-pin| PIN["Peering PIN"]
    HKDF -->|domain: wireguard-psk| PSK["WireGuard PSK"]
    HKDF -->|domain: other-purpose| OTHER["Other keys..."]
`} />

This design means that even if one derived key is compromised, the others remain secure. Changing the domain string produces a completely different key, even from the same secret.

## Peering PIN

When a node wants to join a mesh, it proves knowledge of the mesh secret by presenting a 4-digit PIN derived from the secret. The PIN is computed deterministically:

1. The joining node derives the PIN from the mesh secret
2. The accepting node derives the same PIN from its copy of the secret
3. If the PINs match, the join is authenticated

This avoids transmitting the mesh secret itself over the network during the join handshake.

## What this module does not do

The cryptographic primitives are intentionally scoped to the operations listed above. They do not handle:

- **TLS or mutual TLS** -- handled at the transport layer
- **Disk encryption** -- handled by the storage layer
- **Token signing** -- handled by the auth module

Each layer owns its own cryptographic needs and calls into the core primitives only for the shared building blocks.
