Nauka uses structured logging. Every log line carries context -- not just a timestamp and a message, but the node name, region, zone, and any relevant operational data.

## Output formats

Two formats are available, set via the `logging.format` config key or `NAUKA_LOG_FORMAT` environment variable.

**Text format** (default) -- readable by humans:

```
2026-03-15T14:22:01Z  INFO nauka::fabric  peer connected  node="node-1" region="eu" zone="fsn1"
2026-03-15T14:22:03Z  WARN nauka::fabric  peer unreachable  peer="node-2"
```

**JSON format** -- structured, suitable for log aggregators (Loki, Datadog, etc.):

```json
{"timestamp":"2026-03-15T14:22:01Z","level":"INFO","target":"nauka::fabric","message":"peer connected","node":"node-1","region":"eu","zone":"fsn1"}
```

Use JSON format when running in production with a log aggregation pipeline. Use text format when reading logs interactively or during development.

## Log levels

| Level | When it appears |
|-------|-----------------|
| `error` | Operation failures that require immediate attention |
| `warn` | Degraded state, approaching limits, recoverable issues |
| `info` | Normal operational events: connections, state changes, commands |
| `debug` | Detailed per-operation data, useful for troubleshooting |
| `trace` | Very high volume internal data, rarely needed |

Set the level via config, environment variable, or CLI:

```sh
# Environment variable
NAUKA_LOG_LEVEL=debug nauka hypervisor status

# Config file
# [logging]
# level = "debug"
```

You can also use the standard `RUST_LOG` environment variable for fine-grained control, which overrides the config-based level entirely.

## Per-module filtering

The log level can include module-specific directives for targeted debugging:

```sh
# Global info, but debug for the fabric module
NAUKA_LOG_LEVEL="info,nauka_hypervisor=debug" nauka hypervisor peering
```

This is useful when you need detailed logs from one subsystem without flooding the output with trace data from everything else.

## Global context fields

When Nauka initializes, it registers context fields that appear on every log line:

| Field | Example | Description |
|-------|---------|-------------|
| `node` | `node-1` | This node's name |
| `region` | `eu` | Node's region |
| `zone` | `fsn1` | Node's zone |

These fields are set once at startup. They make it straightforward to filter logs from a specific node when viewing a combined log stream from multiple nodes.

## File logging and rotation

By default, logs go to stderr only. To also write logs to a file, set the `logging.file` config key:

```toml
[logging]
file = "/var/log/nauka/nauka.log"
max_file_size_mb = 50
max_files = 3
```

Log files are rotated daily. Old log files are kept according to the `max_files` setting.

## Dependency noise suppression

Third-party dependencies -- particularly networking and database libraries -- produce high-volume debug and trace logs that are rarely useful during normal operation. Nauka suppresses these to `warn` level by default. The suppressed dependencies include:

- `hyper`, `h2`, `tower` (HTTP stack)
- `tokio`, `mio` (async runtime)
- `redb` (database)
- `rustls`, `reqwest` (TLS and HTTP client)

If you need to debug an issue at the dependency level, use `RUST_LOG` to override:

```sh
RUST_LOG="nauka=debug,hyper=trace" nauka hypervisor peering
```

## Panic handling

If Nauka encounters an unexpected panic, it is routed through the logging system rather than printing raw output to stderr. This means panics appear in log files and JSON output just like other errors, making them visible to log aggregators and alerting systems.

## Warn and error counters

The logging system maintains internal counters for `warn` and `error` level events. These counters can be used to detect unusual error rates without parsing log files.

## CLI logging

CLI commands (such as `nauka hypervisor status`) use a minimal logging configuration at `warn` level with text format. This keeps the CLI output clean -- you only see warnings and errors, not operational info messages.
