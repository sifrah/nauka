Every Nauka API response follows a consistent envelope format. Errors return structured JSON with a machine-readable code, a human-readable message, and an optional suggestion. Success responses use typed envelopes depending on the operation.

## Error format

When a request fails, the API returns a JSON body with three fields:

```json
{
  "code": "RESOURCE_NOT_FOUND",
  "message": "hypervisor 'hv-west-01' not found",
  "suggestion": "Check the name and try again. Use 'nauka hypervisor list' to see available hypervisors."
}
```

- **code** -- a stable, SCREAMING_SNAKE_CASE identifier. Safe to match programmatically. Codes never change between versions.
- **message** -- describes what went wrong in plain language.
- **suggestion** -- when present, tells the operator what to do next.

Every error response also includes an `X-Request-Id` header. Include this value when reporting issues so the server operator can trace the exact request.

## Error codes and HTTP status mapping

Each error code maps to exactly one HTTP status code:

| Code | HTTP | When it happens |
|------|------|----------------|
| `RESOURCE_NOT_FOUND` | 404 | The requested resource does not exist |
| `RESOURCE_ALREADY_EXISTS` | 409 | A resource with that name already exists |
| `VALIDATION_ERROR` | 400 | The request body or parameters are invalid |
| `INVALID_NAME` | 400 | The resource name does not meet naming rules |
| `PERMISSION_DENIED` | 403 | The caller is not authorized for this operation |
| `CONFLICT` | 409 | The operation conflicts with the current resource state |
| `PRECONDITION_FAILED` | 412 | A required precondition is not met |
| `AMBIGUOUS_NAME` | 400 | Multiple resources match the given name |
| `RATE_LIMITED` | 429 | Too many requests in the current window |
| `INTERNAL_ERROR` | 500 | An unexpected server-side error |
| `NOT_IMPLEMENTED` | 501 | The operation is not yet available |
| `DAEMON_UNREACHABLE` | 503 | The Nauka daemon is not responding |
| `TIMEOUT` | 504 | The operation did not complete in time |
| `NETWORK_ERROR` | 502 | A network-level failure occurred |
| `STORAGE_ERROR` | 500 | The embedded storage layer reported an error |

## Retryable errors

Some errors are safe to retry automatically. The following codes are retryable:

- `RATE_LIMITED` -- wait and retry after a brief delay
- `TIMEOUT` -- the operation may succeed on a subsequent attempt
- `NETWORK_ERROR` -- a transient network issue
- `DAEMON_UNREACHABLE` -- the daemon may be restarting

All other error codes indicate a permanent failure that requires operator action.

## Success response envelopes

The API uses different response envelopes depending on the operation type.

**Single resource** (GET, POST create):

The resource object is returned directly as the JSON body, with all fields flattened at the top level.

**List** (GET collection):

```json
{
  "items": [ ... ],
  "count": 3
}
```

The `count` field reflects the number of items in the current response.

**Delete**:

```json
{
  "deleted": true,
  "message": "hypervisor 'hv-west-01' deleted."
}
```

**Action** (POST to a named action like `/hypervisor/init`):

```json
{
  "ok": true,
  "message": "Mesh initialized successfully."
}
```

Action responses may include an optional `data` field with additional structured information when relevant. The `data` field is omitted when there is nothing extra to report.

## CLI error display

When an API error reaches the CLI, it is displayed as a formatted terminal message:

```
  ✖  hypervisor 'hv-west-01' not found
     Check the name and try again. Use 'nauka hypervisor list' to see available hypervisors.
```

The error code and request ID are not shown by default but can be surfaced for debugging purposes.

## Handling errors programmatically

When integrating with the Nauka API, always match on the `code` field rather than parsing the `message` string. Messages may be refined across releases, but error codes are stable and guaranteed not to change.
