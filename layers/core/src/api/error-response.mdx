When an API request fails, the server returns a structured JSON error body. Every error has a machine-readable code, a human-readable message, and a suggestion for what to do next.

## Error response format

```json
{
  "code": "resource_not_found",
  "message": "VPC 'vpc-01HWKZ3P8B0E7D2N5FVYMR4CXJ' does not exist.",
  "suggestion": "Check the ID and try again. Use 'nauka vpc list' to see available VPCs.",
  "request_id": "req-01HWKZ4A9C1F8E3M6GXZNQ5BYK"
}
```

- `code` — a stable, lowercase snake_case identifier for the error type. Safe to match programmatically.
- `message` — a human-readable description of what went wrong.
- `suggestion` — what to do about it. Always present.
- `request_id` — the `X-Request-Id` for this request. Include this when reporting bugs.

## HTTP status code mapping

`NaukaError` variants map to HTTP status codes:

| NaukaError | HTTP status | code |
|-------------|------------|------|
| `NotFound` | 404 | `resource_not_found` |
| `AlreadyExists` | 409 | `already_exists` |
| `ValidationError` | 400 | `validation_error` |
| `MissingArgument` | 400 | `missing_argument` |
| `ConstraintViolation` | 400 | `constraint_violation` |
| `Unauthorized` | 401 | `unauthorized` |
| `Forbidden` | 403 | `forbidden` |
| `RateLimited` | 429 | `rate_limited` |
| `Internal` | 500 | `internal_error` |
| `Unavailable` | 503 | `service_unavailable` |

Internal errors (`500`) never expose implementation details in the message or suggestion. The message is always `"An internal error occurred."` and the suggestion is `"If this persists, contact support with the request ID."`.

## Validation errors

When multiple validation errors occur (e.g., several required fields are missing), they are returned as an array:

```json
{
  "code": "validation_error",
  "message": "The request contains 2 validation errors.",
  "suggestion": "Fix the listed errors and retry.",
  "errors": [
    {
      "field": "name",
      "message": "name is required",
      "suggestion": "Provide a value for the 'name' field."
    },
    {
      "field": "cidr",
      "message": "cidr must be a valid IPv4 CIDR block",
      "suggestion": "Example: 10.0.0.0/16"
    }
  ],
  "request_id": "req-01HWKZ4A9C1F8E3M6GXZNQ5BYK"
}
```

The `errors` array is present only for `validation_error` responses.

## Constraint violation errors

Constraint violations (e.g., `Requires(A, B)` where B is missing) return:

```json
{
  "code": "constraint_violation",
  "message": "The --private flag requires --cidr to be set.",
  "suggestion": "Add --cidr <value> to your request.",
  "request_id": "req-01HWKZ4A9C1F8E3M6GXZNQ5BYK"
}
```

## Handling errors in client code

Match on `code`, not on `message` or HTTP status. Messages can change between versions; codes are stable.

```rust
match error.code.as_str() {
    "resource_not_found" => // 404 - show not found UI
    "already_exists" => // 409 - prompt to use a different name
    "validation_error" => // 400 - highlight the invalid fields
    "unauthorized" => // 401 - redirect to login
    _ => // unexpected error - show generic error with request_id
}
```

## CLI error display

The CLI translates API error responses into terminal-formatted errors using the same `NaukaError` display logic used for local errors:

```
Error: VPC 'vpc-01HWKZ3P8B0E7D2N5FVYMR4CXJ' does not exist.
  Hint: Check the ID and try again. Use 'nauka vpc list' to see available VPCs.
```

The `code` and `request_id` are not shown to the terminal user by default, but are available with `--verbose`.
