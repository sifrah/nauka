Nauka automatically generates REST API routes from the same resource definitions that power the CLI. When a resource is registered with its operations, the corresponding HTTP endpoints are created without any manual route wiring.

## Standard route patterns

For a resource named `hypervisor` with standard CRUD operations, the following routes are generated:

```
GET     /admin/v1/hypervisor          List all hypervisors
POST    /admin/v1/hypervisor          Create a hypervisor
GET     /admin/v1/hypervisor/{id}     Get a specific hypervisor
PATCH   /admin/v1/hypervisor/{id}     Update a hypervisor
DELETE  /admin/v1/hypervisor/{id}     Delete a hypervisor
```

The route prefix (`/admin/v1`) comes from the server configuration. The resource's CLI name is used as the URL segment.

## Custom actions

Resources can define custom actions beyond CRUD. Each action generates a POST route at the collection level:

```
POST    /admin/v1/hypervisor/init     Initialize a new mesh
POST    /admin/v1/hypervisor/join     Join an existing mesh
POST    /admin/v1/hypervisor/ping     Ping a hypervisor
```

Action routes accept a JSON body and return an action response envelope.

## Scoped (nested) routes

Resources that belong to a parent resource generate nested URL paths. The parent's identifier becomes a path parameter:

```
GET     /admin/v1/vpc/{vpc_id}/subnet          List subnets in a VPC
POST    /admin/v1/vpc/{vpc_id}/subnet          Create a subnet in a VPC
```

Multiple levels of nesting are supported. A resource scoped under both an organization and a project produces deeply nested paths:

```
GET     /v1/org/{org_id}/project/{project_id}/vpc          List VPCs
GET     /v1/org/{org_id}/project/{project_id}/vpc/{id}     Get a VPC
```

This URL hierarchy mirrors the logical ownership structure of the resources.

## Request handling

When a request arrives at a generated route, the server:

1. Extracts path parameters (`{id}`, `{vpc_id}`, etc.) from the URL
2. Parses query parameters for list operations (pagination, sorting)
3. Deserializes the JSON request body for create, update, and action operations
4. Validates all field constraints defined on the operation
5. Dispatches to the registered handler

If constraint validation fails, the server returns a `400 Bad Request` with a `VALIDATION_ERROR` code before the handler is called.

## GET and DELETE on the same path

GET and DELETE operations on a single resource share the same URL path (`/{prefix}/{resource}/{id}`) and are differentiated by HTTP method. The route generator merges these into a single path entry with distinct method handlers, avoiding route conflicts.

## OpenAPI specification

The route generator can produce an OpenAPI 3.0 specification from the registered resources. The spec includes:

- All paths and their HTTP methods
- Operation IDs in the format `{resource}_{operation}` (e.g., `hypervisor_list`)
- Summary text from the operation description
- Tag groups by resource name

The specification is served at `GET /openapi.json` when the API server is running. It can also be generated offline for documentation or client code generation.

Example entry from the spec:

```json
{
  "/admin/v1/hypervisor": {
    "get": {
      "summary": "List hypervisors",
      "operationId": "hypervisor_list",
      "tags": ["hypervisor"]
    },
    "post": {
      "summary": "Create a hypervisor",
      "operationId": "hypervisor_create",
      "tags": ["hypervisor"]
    }
  }
}
```

## Route introspection

A route listing function is available to enumerate all registered routes with their HTTP method, URL path, operation name, resource name, and description. This is useful for debugging, documentation generation, or building admin dashboards that need to discover available endpoints.
