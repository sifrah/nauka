The Nauka API server exposes resources over HTTP using a REST interface. It supports two separate listener surfaces (admin and public), applies middleware automatically, and shuts down gracefully when the process receives a termination signal.

## Dual API surfaces

The server can bind two independent listeners:

- **Admin API** -- listens on `127.0.0.1:8443` by default. Intended for local management and operator tooling. All resource routes are prefixed with `/admin/v1`.
- **Public API** -- disabled by default. When enabled, listens on a separate address and exposes a subset of resources under `/v1`. Useful for exposing read-only or tenant-facing endpoints.

If no public resources are registered, the public listener is not started.

## Configuration

The server is configured with the following parameters:

| Parameter | Default | Description |
|-----------|---------|-------------|
| Admin bind address | `127.0.0.1:8443` | Where the admin API listens |
| Public bind address | disabled | Where the public API listens (e.g., `0.0.0.0:443`) |
| Admin prefix | `/admin/v1` | URL prefix for admin routes |
| Public prefix | `/v1` | URL prefix for public routes |
| Rate limit requests | 1000 | Maximum requests per window |
| Rate limit window | 60 seconds | Duration of the rate limit window |

## Built-in endpoints

Every API server automatically registers two endpoints that are always available, regardless of which resources are configured.

### Health check

```
GET /health
```

Returns the server status and version:

```json
{
  "status": "ok",
  "version": "0.3.1"
}
```

This endpoint is designed for load balancer and monitoring health checks. It responds with `200 OK` when the server is running.

### OpenAPI specification

```
GET /openapi.json
```

Returns a minimal OpenAPI 3.0 specification describing the API:

```json
{
  "openapi": "3.0.0",
  "info": {
    "title": "Nauka API",
    "version": "0.3.1"
  },
  "paths": {}
}
```

This endpoint is useful for tooling and API exploration.

## Middleware stack

Every request passes through the following middleware layers before reaching a resource handler:

1. **HTTP tracing** -- structured request/response logging via `tower-http`
2. **X-Request-Id** -- a unique identifier is assigned to every request and returned in the response headers
3. **X-Nauka-Version** -- the server version is attached to every response

See the [middleware documentation](/nauka/layers/core/api/middleware) for details on each layer.

## Graceful shutdown

The server listens for `SIGTERM` and `SIGINT` (Ctrl+C). When either signal is received:

1. The server stops accepting new connections immediately.
2. In-flight requests are allowed to complete.
3. Once all in-flight requests finish, the process exits cleanly.

This ensures that operations like mesh join or hypervisor configuration are not interrupted mid-flight during deployments or restarts.

## Response headers

Every API response includes these headers:

```http
X-Request-Id: req-000000000001
X-Nauka-Version: 0.3.1
```

The request ID is a sequential counter formatted as a zero-padded hexadecimal string prefixed with `req-`. It can be used to correlate requests with server-side logs.
