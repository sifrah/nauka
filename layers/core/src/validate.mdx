Nauka validates all user-supplied input at the boundary -- in the CLI argument parser and in the API request handler -- before any state changes are attempted. The validation rules are defined centrally so they are consistent everywhere. The CLI, the API, and any future interface all enforce the same rules.

## Resource names

Resource names must follow DNS label rules:

| Rule | Constraint |
|------|-----------|
| Length | 3 to 63 characters |
| Characters | Lowercase letters (`a-z`), digits (`0-9`), and hyphens (`-`) only |
| Start | Must start with a lowercase letter |
| End | Must end with a letter or digit |
| Hyphens | No consecutive hyphens (`--`) |

```
Valid:   "production", "web-tier-01", "us-east"
Invalid: "Production"    (uppercase)
Invalid: "web_tier"      (underscore not allowed)
Invalid: "-web"          (starts with hyphen)
Invalid: "web-"          (ends with hyphen)
Invalid: "web--tier"     (consecutive hyphens)
Invalid: "ab"            (too short)
```

These rules match DNS label standards (RFC 1123), which matters because resource names may appear in DNS entries, WireGuard interface names, and Linux network interface names -- all of which have similar constraints.

## Regions and zones

Regions and zones follow simpler rules:

| Rule | Constraint |
|------|-----------|
| Length | 1 to 32 characters |
| Characters | Lowercase letters, digits, and hyphens only |

```
Valid regions:  "eu", "us-east", "htz-fsn1"
Valid zones:    "fsn1", "htz-fsn1-a"
Invalid:        "EU" (uppercase), "us_east" (underscore)
```

## Ports

Ports must be integers in the range 1 to 65535. Port 0 is not valid.

```
Valid:   80, 443, 22, 8080, 51820, 65535
Invalid: 0       (not allowed)
```

Port values can also be provided as strings (e.g., from environment variables), in which case they are parsed and validated in one step.

## Durations

Duration values are parsed from human-readable strings with a numeric value followed by a unit:

| Unit | Meaning | Example |
|------|---------|---------|
| `s` | seconds | `60s` |
| `m` | minutes | `5m` |
| `h` | hours | `2h` |
| `d` | days | `1d` |

```
Valid:   "5m", "60s", "2h", "1d", "30s"
Invalid: "0s"         (zero duration)
Invalid: "30"         (no unit)
Invalid: "30sec"      (unknown unit)
Invalid: "banana"     (not a number)
```

Durations are used throughout the configuration for intervals and timeouts, such as `health_check_interval = "60s"` and `join_timeout = "5m"`.

## CIDR blocks

IPv4 CIDR blocks are validated for correct syntax and network alignment:

| Rule | Constraint |
|------|-----------|
| Format | `A.B.C.D/N` |
| Octets | Each must be 0-255 |
| Prefix length | 0-32 |
| Host bits | Must be zero (network address only) |

```
Valid:   "10.0.0.0/8", "192.168.1.0/24", "172.16.0.0/12"
Invalid: "10.0.0.1/8"   (host bits set -- did you mean 10.0.0.0/8?)
Invalid: "256.0.0.0/8"  (invalid octet)
Invalid: "10.0.0.0/33"  (prefix too long)
```

When host bits are set, the error message suggests the correct network address, making it easy to fix the input.

## Labels

Labels are key-value pairs in `key=value` format, used for tagging resources:

| Rule | Constraint |
|------|-----------|
| Key length | 1 to 63 characters |
| Key characters | Letters, digits, hyphens, dots, underscores |
| Key start | Must start with a letter |
| Value length | 0 to 63 characters |

```
Valid:   "env=production", "team=platform", "version=2.1"
Invalid: "=value"          (empty key)
Invalid: "1key=value"      (key starts with digit)
```

## Resource sizes

Storage and compute sizes have platform-defined bounds:

| Resource | Minimum | Maximum |
|----------|---------|---------|
| Disk size | 1 GB | 65,536 GB (64 TB) |
| Memory | 128 MB | 1,048,576 MB (1 TB) |
| vCPUs | 1 | 256 |

## Validation errors

All validation functions produce a clear error that names the field and the rule that failed:

```
Error: invalid name 'Web-Tier': invalid character 'W' -- only lowercase letters, digits, and hyphens allowed
```

```
Error: invalid duration 'banana': must be a number followed by s/m/h/d (e.g., 30m, 2h, 7d)
```

These errors are caught at the entry point, before any state changes are made, so the system is never left in a partially modified state due to bad input.
