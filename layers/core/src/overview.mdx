The Core layer is the foundation of the Nauka platform. It defines the shared rules and structures that every other layer depends on: how nodes are addressed, how resources are identified, how configuration is loaded, how errors are reported, and how input is validated.

As a platform engineer, you will not interact with the Core layer directly. Instead, you interact with it indirectly every time you run a CLI command, read an error message, configure the platform, or inspect a resource ID. Understanding what Core provides helps you understand why the platform behaves the way it does.

Every layer imports from Core. Nothing in Core imports from any other layer. This makes Core the stable base that never changes in response to higher-layer decisions.

## What Core provides

| Capability | What it means for you |
|-----------|----------------------|
| [Addressing](/nauka/layers/core/addressing) | Every node gets a deterministic IPv6 address derived from its WireGuard key -- no DHCP, no address pools |
| [Cryptography](/nauka/layers/core/crypto) | WireGuard keypairs and mesh secrets are generated and managed securely |
| [Resource IDs](/nauka/layers/core/id) | Every resource has a typed, sortable ID like `hv-01J5A3K7G8MN2P4Q6R9S0T1V2W` |
| [Configuration](/nauka/layers/core/config) | A clear priority chain: CLI flags > environment variables > config file > defaults |
| [Validation](/nauka/layers/core/validate) | Names, ports, durations, and regions are validated consistently at every entry point |
| [Error model](/nauka/layers/core/error) | Every error has a code, a message, and a suggestion telling you what to do next |
| [Versioning](/nauka/layers/core/version) | Semver with release channels, compatibility checking between nodes, and rollback support |
| [Logging](/nauka/layers/core/logging) | Structured logging in text or JSON format with file rotation and dependency noise suppression |
| [Process utilities](/nauka/layers/core/process) | Standard directory layout and socket paths under `~/.nauka/` |
| [Transport](/nauka/layers/core/transport) | Unix domain socket protocol for CLI-to-daemon communication |

## Design principles

**Errors are actionable.** Every error carries a suggestion -- a human-readable hint about what to do next. You will never see a bare error code without guidance on how to resolve it.

**Types encode intent.** A hypervisor ID (`hv-...`) and a VPC ID (`vpc-...`) are distinct types. You cannot accidentally pass one where the other is expected. The type system catches mistakes before they reach production.

**Validation at the boundary.** All user-supplied input is validated the moment it enters the system -- in the CLI parser or in the API handler. Invalid input is rejected with a clear error before any state changes are made.

**No I/O in Core.** Core defines types and validation logic. It never reads files, opens sockets, or spawns threads on its own. This keeps Core simple, testable, and free of side effects.
