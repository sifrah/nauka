The transport module defines how the Nauka CLI and other local clients communicate with a running Nauka process. All communication happens over a Unix domain socket using a simple, inspectable wire format.

## Why Unix domain sockets?

Unix domain sockets are the right choice for local process-to-process communication:

- **No network overhead** -- faster than loopback TCP for local calls
- **File permissions** -- socket access is controlled by filesystem permissions, not port numbers or firewalls
- **No port conflicts** -- no risk of colliding with other services on the host
- **Clean lifecycle** -- the socket file is created when the process starts and removed when it exits

The socket is located at `~/.nauka/control.sock` by default (see [Process and Directory Layout](/nauka/layers/core/process)).

## Wire protocol

Messages are length-prefixed JSON:

<Mermaid chart={`
sequenceDiagram
    participant CLI
    participant Daemon as Nauka Process
    CLI->>Daemon: [4 bytes: length] [JSON request]
    Daemon->>CLI: [4 bytes: length] [JSON response]
    CLI->>CLI: Close connection
`} />

The 4-byte header is a big-endian unsigned 32-bit integer encoding the byte length of the JSON body that follows. The receiver reads the header first, allocates a buffer of that size, then reads the body.

The maximum message size is **1 MB**. Messages larger than this are rejected.

## Request format

Every request specifies a resource kind, an operation, and optional parameters:

```json
{
  "kind": "hypervisor",
  "operation": "status",
  "name": "my-node",
  "fields": {}
}
```

| Field | Description |
|-------|-------------|
| `kind` | The resource type being operated on (`hypervisor`, `vpc`, `fabric`, etc.) |
| `operation` | The action to perform (`create`, `list`, `get`, `delete`, or a custom action) |
| `name` | Resource name or ID (for operations that target a specific resource) |
| `scope` | Scope values for hierarchical resources (e.g., which org or VPC) |
| `fields` | Additional parameters from CLI flags |

## Response format

Responses indicate success or failure with structured data:

**Success:**

```json
{
  "ok": true,
  "data": {
    "name": "my-mesh",
    "status": "active",
    "peers": 3
  }
}
```

**Error:**

```json
{
  "ok": false,
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "hypervisor 'staging' not found",
    "suggestion": "Run 'nauka hypervisor list' to see available nodes"
  }
}
```

The error in the response follows the standard [error model](/nauka/layers/core/error), so error handling is consistent whether the error originates locally in the CLI or remotely from the daemon.

## Request routing

The Nauka process uses a router to dispatch incoming requests to the appropriate handler based on the `kind` field. Each layer registers its handlers during startup. When a request arrives:

1. The router looks up the handler for the request's `kind`
2. If found, the request is passed to the handler
3. If no handler is registered for that kind, a `RESOURCE_NOT_FOUND` error is returned with a list of available resource kinds

## Socket permissions

The control socket is created with restrictive permissions (mode `0600`) so that only the socket owner can connect. This prevents other users on the same host from sending commands to the Nauka process.

## Inspecting traffic

Because the wire format is JSON (after the 4-byte length header), you can inspect socket traffic during debugging using standard tools:

```sh
socat -v UNIX-CONNECT:~/.nauka/control.sock -
```

This is useful for verifying that requests are being formatted correctly or for building custom integrations that communicate directly with the Nauka process.
