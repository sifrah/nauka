The service module manages WireGuard as a persistent systemd service. It handles everything from installing `wireguard-tools` if missing, to writing the WireGuard configuration, registering the systemd unit, and providing lifecycle operations. The service survives reboots and runs independently of the `nauka` CLI process.

This module re-exports the canonical implementation from the WireGuard backend. Both import paths are equivalent.

## Files on disk

| Path | Purpose | Permissions |
|---|---|---|
| `/etc/wireguard/nauka0.conf` | WireGuard interface configuration | `0600` (root only) |
| `/etc/systemd/system/nauka-wg.service` | systemd unit file | Standard |

The WireGuard configuration file is restricted to root-only read access because it contains the node's private key.

## Automatic package installation

Before setting up WireGuard, Nauka checks whether `wg-quick` is available on the system. If not, it automatically installs `wireguard-tools` using the detected package manager:

| Package manager | Distribution |
|---|---|
| `apt-get` | Debian, Ubuntu (runs `apt-get update` first) |
| `dnf` | Fedora, RHEL 8+ |
| `yum` | CentOS, older RHEL |
| `pacman` | Arch Linux |
| `apk` | Alpine Linux |
| `zypper` | openSUSE |

If no supported package manager is found, Nauka returns an error asking the operator to install `wireguard-tools` manually.

## WireGuard configuration

The configuration file `/etc/wireguard/nauka0.conf` is generated from the fabric state. A typical configuration looks like:

```ini
[Interface]
PrivateKey = <this node's WG private key>
ListenPort = 51820
Address = fd12:3456:789a::1/128

[Peer]
PublicKey = <peer's public key>
AllowedIPs = fd12:3456:789a::2/128
PersistentKeepalive = 25
Endpoint = 203.0.113.20:51820
```

Key points about the configuration:

- Each known peer gets its own `[Peer]` section
- `AllowedIPs` is set to the peer's single mesh IPv6 address (`/128`), so only mesh traffic is routed through the tunnel -- not the peer's general internet traffic
- `PersistentKeepalive` is set to 25 seconds, ensuring regular handshakes even without application traffic (important for health monitoring)
- `Endpoint` is included only when known (it may be omitted for peers behind NAT that have not yet connected)

## The systemd unit

The `nauka-wg` service is a `Type=oneshot` unit with `RemainAfterExit=yes`:

```ini
[Unit]
Description=Nauka WireGuard Mesh
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/bin/wg-quick up /etc/wireguard/nauka0.conf
ExecStop=/usr/bin/wg-quick down /etc/wireguard/nauka0.conf

[Install]
WantedBy=multi-user.target
```

This means:
- The service waits for network to be online before starting
- `wg-quick up` creates the interface and then the service is considered "active"
- `wg-quick down` tears down the interface on stop
- The service starts automatically on boot when enabled

## Service lifecycle operations

| Operation | Effect |
|---|---|
| Install | Write config + unit file, run `systemctl daemon-reload` |
| Enable and start | `systemctl enable --now nauka-wg` |
| Start | `systemctl start nauka-wg` |
| Stop | `systemctl stop nauka-wg` |
| Uninstall | Disable service, remove unit + config files, reload daemon |
| Update config | Rewrite `nauka0.conf`, apply with `wg syncconf` (no teardown); falls back to service restart if `syncconf` fails |

## Live configuration reload

When a peer joins or leaves and the WireGuard configuration needs updating, Nauka uses `wg syncconf nauka0 /etc/wireguard/nauka0.conf` to apply changes without tearing down the interface. This is a non-disruptive operation -- existing tunnels continue running while the new peer is added or removed.

If `wg syncconf` fails (which can happen in edge cases), the service is fully restarted as a fallback. This guarantees peers are always in sync with the state database.

## Permissions

All operations that write to `/etc/wireguard/` or invoke `systemctl` require root privileges. Nauka expects to be run as root on hypervisor nodes. The WireGuard configuration file is written with `0600` permissions to protect the private key.
