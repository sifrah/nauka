Every node in a Nauka mesh maintains a local list of all other nodes it knows about. This peer list is the node's view of the mesh: who else is in it, how to reach them over WireGuard, and whether they are currently reachable. The peer list drives WireGuard configuration -- every entry in the list corresponds to a `[Peer]` section in `/etc/wireguard/nauka0.conf`.

## What a peer record contains

Each remote node is represented by a peer record with the following properties:

| Property | Description |
|---|---|
| Peer ID | Unique identifier for this peer (prefixed `node-`) |
| Name | Human-readable name (e.g., `node-2`) |
| Region | Region the peer is located in |
| Zone | Availability zone within the region |
| WireGuard public key | The peer's public key, used to establish the encrypted tunnel |
| WireGuard port | UDP listen port on the peer (default 51820) |
| Endpoint | Public IP:port used to reach the peer over the internet |
| Mesh IPv6 | The peer's unique address within the mesh prefix |
| Status | Current reachability status (see below) |
| Last handshake | Unix timestamp of the last successful WireGuard handshake |
| Added at | Unix timestamp of when this peer was added to the list |

## Peer status

A peer is always in one of three states:

| Status | Meaning |
|---|---|
| **Active** | The WireGuard handshake completed within the last 5 minutes (300 seconds). The tunnel is healthy and traffic can flow. |
| **Unreachable** | No successful handshake in the last 5 minutes. The peer may be down, its endpoint may have changed, or there may be a network issue between the two nodes. |
| **Removed** | The peer has explicitly left the mesh (via `nauka hypervisor leave`). It should no longer receive traffic. |

The 300-second threshold is chosen to accommodate WireGuard's normal behavior. WireGuard initiates a handshake approximately every 2 minutes when there is active traffic between peers. A 5-minute window without any handshake is a strong signal that the peer is genuinely unreachable, not just experiencing a momentary gap in traffic.

Status transitions are managed by the health check loop, which runs every 30 seconds during peering mode and updates peer statuses based on WireGuard handshake timestamps.

## How the peer list is populated

Peers are added to the list through two mechanisms:

1. **During join** -- When a node joins the mesh, the accepting peer sends back the complete list of all known peers as part of the join response. The joining node initializes its peer list from this snapshot, including the accepting node itself.
2. **Via peer announcement** -- After a join completes, the accepting node broadcasts a peer announcement to all existing members, telling them about the new node. Each existing node adds the new peer to its own list.

## Uniqueness constraints

The peer list enforces two uniqueness constraints:

- **No duplicate public keys** -- Each WireGuard public key can appear only once. This prevents the same node from being added twice.
- **No duplicate names** -- Each node name must be unique. This prevents confusion when looking up peers by name.

If a node leaves and rejoins with the same name but a different key (which is the normal case, since keys are regenerated), the stale entry is replaced with the new one during the announce handling.

## Peer removal

Peers can be removed from the list by name or by public key. When a node runs `nauka hypervisor leave`, it broadcasts a removal message to all known peers. Each receiving peer removes the entry from its list, updates its WireGuard configuration, and persists the change.

If a peer goes down without running the leave procedure (e.g., hardware failure), it will be marked as **Unreachable** by the health check loop on all other nodes. The entry remains in the peer list but no traffic is routed to it.

## Persistence

The peer list is part of the fabric state and is persisted to the embedded database on every change. If the node restarts, it reloads the full peer list and reconfigures WireGuard from it. The persisted state is the source of truth -- any manual changes to the WireGuard configuration will be overwritten on the next service restart.
