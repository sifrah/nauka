Fabric state is the complete, persisted snapshot of a node's mesh membership. It is the single source of truth for what mesh this node belongs to, who this node is, what peers it knows about, and what shared secret authenticates new joiners. Every `nauka hypervisor` command reads from or writes to this state.

## What the state contains

| Component | Description |
|---|---|
| Mesh identity | The mesh's unique ID and `/48` IPv6 prefix -- shared by all mesh members |
| Hypervisor identity | This node's ID, name, region, zone, WireGuard keys, port, endpoint, and mesh IPv6 address |
| Mesh secret | The shared secret (`syf_sk_...`) used to derive the PIN for authenticating new joins |
| Peer list | All known remote peers with their WireGuard keys, endpoints, mesh addresses, and current status |
| Network mode | The network backend this node uses (`wireguard`, `direct`, or `mock`) |

All five components must be present for a node to be considered initialized. If any are missing or corrupted, the node is in a partial state and most commands will refuse to run.

## Where the state is stored

The fabric state is stored in the embedded database under the table `fabric` with the key `state`. The database file is located in the Nauka data directory. Storage uses the state layer, which provides atomic writes and crash safety through the embedded `redb` database engine.

The key structure is intentionally simple -- there is exactly one fabric state record per node. There is no history, no versioning, and no multi-record complexity. The current state is the complete truth.

## State lifecycle

### Created during init or join

When you run `nauka hypervisor init`, the state is created with:
- A freshly generated mesh identity
- A freshly generated hypervisor identity (including WireGuard keypair)
- The mesh secret
- An empty peer list
- The selected network mode

When you run `nauka hypervisor join`, the state is created with:
- The mesh identity received from the accepting node
- A locally generated hypervisor identity
- The mesh secret received from the accepting node
- The peer list received from the accepting node (pre-populated with all existing members)
- The selected network mode

### Updated during operation

The state is updated whenever:
- A new peer joins (added to the peer list)
- A peer leaves (removed from the peer list)
- A health check detects a status change (peer marked active or unreachable)

Each update is an atomic write -- the entire state is serialized and written as a single database transaction.

### Deleted during leave

When you run `nauka hypervisor leave`, the state is deleted after the network is torn down. Once deleted, the node is as if it was never part of any mesh and can be re-initialized or joined to a different mesh.

## Consistency guarantees

Because the state is written atomically through the embedded database, there is no risk of partial writes. A save either fully commits or leaves the previous value intact. This means:

- A crash during `init` or `join` leaves the database either empty (node not initialized) or with the previous valid state -- never with a corrupted half-written record.
- A crash during a peer list update leaves the previous peer list intact. The next health check or announce cycle will reconcile any missed changes.

## Relationship between state and WireGuard

The fabric state is the definitive record. The WireGuard configuration file (`/etc/wireguard/nauka0.conf`) and the live WireGuard interface are derived from it. On every service start, WireGuard is configured from the persisted state. If you manually edit the WireGuard configuration or use `wg` commands directly, those changes will be overwritten the next time the service restarts or a peer change occurs.

Always use `nauka hypervisor` commands to modify mesh membership. Direct WireGuard modifications are not persisted and will be lost.

## Checking state

You can check whether state exists and inspect its contents:

```sh
# Check if the node is initialized
nauka hypervisor status

# If not initialized, you will see:
# Error: not initialized. Run 'nauka hypervisor init' first.
```

The `status` command loads the fabric state and reports all key information: node identity, mesh details, peer counts, service status, and WireGuard interface health.
