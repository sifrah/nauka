Fabric operations are the entry points for every mesh lifecycle action. When you run a `nauka hypervisor` command, it calls one of the operations described on this page. Each operation orchestrates the lower-level subsystems -- identity generation, network backend, WireGuard configuration, state persistence -- into a single coherent workflow.

Nauka is a CLI orchestrator, not a daemon. Every operation configures the system (systemd services, WireGuard interfaces, persistent state) and then exits. The services it installs continue running independently.

## Initializing a new mesh

The `init` operation creates the first node in a new mesh. It is the starting point for every Nauka deployment.

```sh
nauka hypervisor init --name node-1 --region eu --zone fsn1
```

When you run init, the following steps happen in order:

1. **Pre-flight check** -- Nauka verifies that no mesh state already exists on this machine. If the node was previously initialized, you must run `nauka hypervisor leave` first.
2. **Network backend installation** -- The selected network backend (WireGuard by default) is installed. For WireGuard mode, this means ensuring `wireguard-tools` is present on the system, installing it automatically via the system package manager if needed.
3. **Identity generation** -- A new mesh identity is created with a unique ID and a randomly generated `/48` ULA IPv6 prefix (in the `fd00::/8` range). A WireGuard keypair is generated, and the node's mesh IPv6 address is derived deterministically from the prefix combined with the public key.
4. **Network setup** -- The WireGuard configuration file is written to `/etc/wireguard/nauka0.conf`, the `nauka-wg` systemd service unit is installed, and the service is started. The interface `nauka0` comes up with the node's mesh IPv6 address.
5. **State persistence** -- The complete fabric state (mesh identity, hypervisor identity, mesh secret, empty peer list) is saved to the embedded database.
6. **PIN derivation** -- A short numeric PIN is derived from the mesh secret. This PIN is displayed to the operator and is required when other nodes join the mesh.

The output includes the mesh secret (partially masked) and the PIN. The mesh secret should be stored securely -- anyone who can derive the PIN from the secret can join the mesh.

### Init parameters

| Parameter | CLI flag | Default | Description |
|---|---|---|---|
| Node name | `--name` | hostname | Unique name within the mesh (max 63 chars, lowercase alphanumeric + hyphens) |
| Region | `--region` | required | Logical region label (e.g., `eu`, `us-east`) |
| Zone | `--zone` | required | Availability zone within the region (e.g., `fsn1`, `nbg1`) |
| Port | `--port` | `51820` | WireGuard listen port. Peering uses port+1, announce uses port+2 |
| Network mode | `--mode` | `wireguard` | Network backend: `wireguard`, `direct`, or `mock` |
| Fabric interface | `--interface` | auto | Physical network interface for fabric traffic |
| Endpoint | `--endpoint` | auto-detected | Public IP:port to advertise to peers |

All nodes in a mesh should use the same network mode.

## Joining an existing mesh

The `join` operation connects a new node to an existing mesh by contacting a node that is already a member and has peering mode active.

```sh
nauka hypervisor join --target 203.0.113.10 --pin G7CCZX --name node-2 --region eu --zone nbg1
```

The join flow:

1. **Pre-flight check** -- Verifies no existing state on this machine.
2. **Network backend installation** -- Ensures the required tools are available.
3. **TLS connection** -- Connects to the target's peering port (WireGuard port + 1, default 51821) over TCP with TLS encryption.
4. **Join request** -- Sends a request containing the node's name, region, zone, WireGuard public key, listen port, and PIN. If no endpoint is specified, the target detects it from the TCP connection's source IP.
5. **Join response** -- The target validates the PIN, then responds with the mesh secret, the `/48` prefix, the mesh ID, its own peer information, and the complete list of all other known peers.
6. **Local setup** -- The joining node derives its mesh IPv6 address, builds its WireGuard configuration with all received peers, installs and starts the `nauka-wg` service, and persists the complete fabric state.
7. **Peer notification** -- The target broadcasts the new peer's information to all existing mesh members via the announce protocol, so they update their WireGuard configurations.

The target must be a running Nauka hypervisor with peering mode active (`nauka hypervisor peering`). The PIN is the one displayed when the mesh was initialized.

### Concurrent joins

Multiple nodes can join the same mesh simultaneously. The peering server processes joins sequentially, and each joiner receives the full peer list known at that point. Three mechanisms ensure all nodes eventually discover each other:

- **Immediate broadcast** -- When a join is accepted, the target announces the new peer to all existing members. This is best-effort; some peers may not yet have their announce listener running.
- **Persistent announce listener** -- Each node that runs peering mode listens on `wg_port + 2` for incoming peer announcements.
- **Periodic reconciliation** -- Every 30 seconds, the peering node re-reads the peer list and re-announces any peers added within the last 60 seconds to all other peers. This is idempotent (known peers are skipped) and guarantees full mesh convergence even when joins overlap.

The mesh reaches full connectivity within approximately 60 seconds (two reconciliation cycles) of the last join, regardless of join ordering.

## Peering mode

The `peering` command activates the node to accept join requests and run background mesh maintenance.

```sh
nauka hypervisor peering
```

This starts four concurrent background tasks:

| Task | Port | Description |
|---|---|---|
| Peering listener | `wg_port + 1` (51821) | Accepts TLS connections from joining nodes, validates PINs, sends mesh credentials |
| Announce listener | `wg_port + 2` (51822) | Receives peer announcements and removals from other nodes over TLS |
| Health check loop | (internal) | Polls WireGuard handshake timestamps every 30s, updates peer status |
| Reconciliation loop | (internal) | Re-announces recent peers every 30s for convergence |

The peering listener opens a fresh database connection for each incoming join request, so one slow operation does not block others. All listeners bind to `[::]` (all interfaces, both IPv4 and IPv6).

### Port allocation summary

| Purpose | Default Port | Derivation |
|---|---|---|
| WireGuard data plane (UDP) | 51820 | `wg_port` |
| Peering listener (TCP/TLS) | 51821 | `wg_port + 1` |
| Announce listener (TCP/TLS) | 51822 | `wg_port + 2` |

If you change the WireGuard port via `--port`, the peering and announce ports shift accordingly.

## Checking mesh status

```sh
nauka hypervisor status
```

The status operation loads fabric state, queries the WireGuard interface, and checks the systemd service. It reports:

- Node name, ID, region, and zone
- Mesh IPv6 address
- Overall state: **available** (service running and interface up), **stopped** (service installed but not active), or **not installed**
- Whether the `nauka-wg` systemd service is installed and active
- Whether the `nauka0` interface is up
- WireGuard listen port, peer count, and traffic counters (RX/TX bytes)
- Count of total peers vs. active peers

## Starting and stopping the network

```sh
nauka hypervisor start
nauka hypervisor stop
```

`start` enables the WireGuard service via the network backend. It is idempotent -- if the service is already running, nothing happens. `stop` brings down the WireGuard interface and closes all peer tunnels, but preserves fabric state so you can `start` again to reconnect.

Both operations require that the node has been initialized.

## Leaving the mesh

```sh
nauka hypervisor leave
```

The `leave` operation permanently removes this node from the mesh:

1. **Notify peers** -- Broadcasts a `PeerRemove` message to all known peers over TLS. Each receiving peer removes this node from its WireGuard configuration and peer list. The broadcast retries up to 4 times with 3-second delays between attempts to handle transient failures.
2. **Tear down network** -- The network backend stops the service, removes configuration files, and uninstalls the systemd unit.
3. **Delete state** -- The fabric state is deleted from the embedded database.

After leaving, the node is clean and can be re-initialized or joined to a different mesh. Peers that receive the removal notification immediately clean up. Peers that miss it (e.g., they were offline) will detect the departed node as unreachable through the health check loop.
