The `ops` module is where the work actually happens. Every high-level `nauka hv` command ultimately calls a function here. Ops orchestrates the other fabric sub-modules — identity, WireGuard, systemd, peering, state — into coherent multi-step workflows.

## InitConfig

`init` and `join` accept an `InitConfig` struct that consolidates all the parameters that influence how the node is set up:

| Field | Flag | Description |
|---|---|---|
| `name` | `--name` | Unique node name within the mesh |
| `region` | `--region` | Region identifier (e.g., `us-east`) |
| `zone` | `--zone` | Availability zone (e.g., `us-east-1a`) |
| `mode` | `--mode` | Network backend: `wireguard` (default), `direct`, or `mock` |
| `interface` | `--interface` | WireGuard interface name (default: `nauka0`) |
| `endpoint` | `--endpoint` | This node's public endpoint (`host:port`) advertised to peers |

The `--mode` flag selects the [network backend](/nauka/layers/hypervisor/fabric/backend). All nodes in a cluster should use the same mode.

The `--endpoint` flag is optional. If omitted, Nauka detects the public IP automatically. Set it explicitly when running behind NAT or when you want to control which address peers connect to.

## init

`init` bootstraps the first node in a new mesh.

**What it does:**

1. Generates a `MeshIdentity` (UUID + `/48` prefix) via `create_mesh()`
2. Generates a `HypervisorIdentity` (WG keypair, assigns mesh IPv6) via `create_hypervisor()`
3. Selects the network backend from `InitConfig.mode` via `create_backend()`
4. Installs the backend service via the [service module](/nauka/layers/hypervisor/fabric/service)
5. Starts the service
6. Persists `FabricState` (including `network_mode`) to `~/.nauka/fabric.redb`
7. Derives a short PIN from the mesh secret for use during peer joins

**Usage:**

```sh
nauka hv init --name web-01 --region us-east --zone us-east-1a
nauka hv init --name web-01 --region us-east --zone us-east-1a --mode direct
nauka hv init --name web-01 --region us-east --zone us-east-1a --endpoint 203.0.113.10:51820
# Mesh secret: syf_sk_AbC123...
# PIN: 7842
```

If any step fails, init cleans up (stops and uninstalls the service, removes state) so you can safely retry.

## join

`join` connects a new node to an existing mesh.

**What it does:**

1. Connects to the target's peering port (WG port + 1) via TCP
2. Sends a `JoinRequest` with name, region, zone, WG pubkey, and PIN
3. Receives a `JoinResponse` containing the mesh secret, the `/48` prefix, and the full peer list
4. Creates a `HypervisorIdentity` for this node using the received mesh data
5. Installs and starts the WireGuard service with all received peers pre-configured
6. Persists `FabricState`

**Usage:**

```sh
nauka hv join --target 203.0.113.10 --name web-02 --region us-east --zone us-east-1a
```

The target must be a running Nauka hypervisor. You need the PIN displayed when the mesh was initialized.

## listen_for_peers

`listen_for_peers` starts the peering mode for this node. It spawns three background tasks and then runs until cancelled:

1. **Peering listener** — TCP server on `wg_port + 1` (default 51821) that accepts incoming `join` requests. For each request it opens a fresh DB connection, validates the PIN, sends the mesh secret and peer list to the joiner, and calls `broadcast_new_peer()` to notify existing peers.
2. **Health check loop** — polls WireGuard handshake timestamps every 30 seconds and updates peer status (`Active` / `Unreachable`). See [Health Checks](/layers/hypervisor/fabric/health).
3. **Announce listener** — TCP server on `wg_port + 2` (default 51822) that receives `PeerAnnounce` messages from other nodes and adds newly joined peers to this node's WireGuard config. See [Peer Announcement](/layers/hypervisor/fabric/announce).

**Port allocation summary:**

| Purpose | Port |
|---|---|
| WireGuard data plane | 51820 (`wg_port`) |
| Peering listener | 51821 (`wg_port + 1`) |
| Announce listener | 51822 (`wg_port + 2`) |

All three tasks run concurrently. The per-request database open on the peering listener is intentional — it prevents one slow join from blocking other operations.

## status

`status` loads `FabricState`, queries the WireGuard interface, and checks the systemd service state. It returns:

- Mesh ID and prefix
- This node's name, region, zone, mesh IPv6
- Service status (running / stopped)
- WireGuard listen port, peer count, RX/TX bytes
- List of peers with their current status

```sh
nauka hv status
```

## start

`start` enables and starts the `wg-quick@nauka0` systemd service. If the service is already running, the command succeeds without doing anything. State must exist (i.e. the node must have been initialized) for start to work.

```sh
nauka hv start
```

## stop

`stop` stops the `wg-quick@nauka0` service. The WireGuard interface is brought down, and all peer tunnels are closed. State is preserved — you can `start` again to reconnect.

```sh
nauka hv stop
```

## leave

`leave` permanently removes this node from the mesh.

**What it does:**

1. Stops the WireGuard service
2. Uninstalls the service and config files
3. Removes the `nauka0` interface if it still exists
4. Deletes `FabricState` from the database

After `leave`, the node is clean. Other nodes will detect it as `Unreachable` and eventually mark it `Removed`. To rejoin, run `nauka hv join` again.

```sh
nauka hv leave
```
