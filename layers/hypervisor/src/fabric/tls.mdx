All control-plane connections in the Nauka mesh -- peering joins and peer announcements -- are encrypted with TLS. This page explains the trust model, why it uses self-signed certificates, and how authentication actually works.

## The security model

Nauka's peering protocol separates encryption from authentication:

| Layer | Responsibility |
|---|---|
| **TLS** | Encrypts all traffic between nodes. An eavesdropper on the network sees only encrypted bytes. |
| **PIN** | Authenticates the joining node. Proves the operator has been given access by someone who controls the mesh. |

This separation is deliberate. TLS provides confidentiality (nobody can read the traffic), while the PIN provides authorization (only approved nodes can join). Neither layer alone is sufficient, but together they provide a practical security model for bare-metal infrastructure.

## Self-signed certificates

The peering server generates a fresh self-signed TLS certificate each time it starts. There is no certificate authority (CA), no certificate pinning, and no DNS name to verify. The certificate:

- Is issued for the hostname `nauka-peering`
- Is generated at runtime using a fresh key pair
- Lives only in memory -- it is never written to disk
- Is different on every startup

Because the certificate is ephemeral and self-signed, it cannot be used for identity verification. Its only purpose is to establish an encrypted channel.

## Trust on First Use (TOFU)

When a joining node connects to the peering server, the TLS client deliberately skips certificate verification. It accepts whatever certificate the server presents. This is the Trust on First Use (TOFU) model.

This might sound insecure, but consider the threat model:

- In a new mesh, there is no pre-existing trust anchor to verify against. No CA has issued a certificate for a server that was just created seconds ago.
- The TLS certificate's only job is to prevent passive eavesdropping. It ensures the mesh secret and PIN cannot be captured in transit.
- Active authentication is handled by the PIN, which is a shared secret established out-of-band (displayed on the terminal during `nauka hypervisor init` and communicated to the joining operator through a trusted channel).

An attacker who can observe network traffic sees only encrypted bytes. An attacker who can actively intercept the connection (man-in-the-middle) would need the PIN to complete the join -- without it, no mesh credentials are disclosed.

## Why not a traditional PKI?

A traditional PKI would require one of:

- **A pre-shared CA certificate** -- which itself needs to be distributed out-of-band before the first join, creating a chicken-and-egg problem no easier than sharing a PIN.
- **A certificate from a public CA** -- which requires a domain name, internet-accessible server, and ACME challenge infrastructure. This is impractical for bare-metal servers in private networks.

The PIN-based model achieves equivalent practical security with significantly less operational overhead. The PIN is short, human-typeable, and can be communicated over any trusted channel (phone call, chat message, in-person).

## TLS for announce connections

Peer announcements (sent on `wg_port + 2`) also use TLS with the same TOFU model. Since announce traffic travels over the WireGuard mesh (using mesh IPv6 addresses), it already benefits from WireGuard's encryption. The TLS layer adds defense-in-depth against injection attacks by adversaries who might have access to the mesh network.

## Practical implications for operators

- **No certificates to manage** -- Nauka handles all certificate generation internally. There are no certificate files to deploy, rotate, or renew.
- **No trust stores to configure** -- The TOFU model means no CA bundles or trust anchors need to be installed.
- **PIN is the critical secret** -- Protect the PIN as you would any authentication credential. Anyone with the PIN and network access to the peering port can join the mesh.
- **PIN rate limiting** -- The peering server limits PIN brute-force attempts to 5 failures per IP, with a 10-minute block after the threshold is exceeded.
