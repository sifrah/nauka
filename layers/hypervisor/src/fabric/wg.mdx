The WireGuard interface module manages the `nauka0` network interface directly through the standard `wg` and `ip` command-line tools. It does not use netlink or any C bindings -- all operations shell out to standard system commands, keeping the implementation portable and auditable.

This module re-exports the canonical implementation from the WireGuard backend. Both import paths are equivalent.

## The nauka0 interface

All Nauka mesh traffic flows through a single WireGuard interface named `nauka0`. Every node in the mesh has this interface. The name is fixed and you will see it in standard system tools:

```sh
ip link show nauka0
# 5: nauka0: <POINTOPOINT,NOARP,UP,LOWER_UP> mtu 1420 ...

wg show nauka0
# interface: nauka0
#   public key: ...
#   listening port: 51820
```

## Interface creation

When a node initializes or joins a mesh, the interface is created through the following steps:

1. **Create the kernel interface** -- `ip link add dev nauka0 type wireguard`
2. **Configure WireGuard** -- The private key is piped through stdin to `wg set nauka0 private-key /dev/stdin listen-port <port>`. The key is never visible in the process argument list.
3. **Assign the mesh address** -- `ip -6 addr add <mesh_ipv6>/128 dev nauka0`
4. **Bring the interface up** -- `ip link set up dev nauka0`

In practice, the service module uses `wg-quick` (which handles these steps internally) for the initial setup. The direct interface commands are used for runtime modifications when the interface is already running.

## Peer management

### Adding a peer

When a new node joins the mesh, each existing node adds it to its WireGuard interface:

```sh
wg set nauka0 peer <pubkey> allowed-ips <mesh_ipv6>/128 persistent-keepalive 25 endpoint <host:port>
ip -6 route add <mesh_ipv6>/128 dev nauka0
```

The `allowed-ips` field restricts the peer to its single mesh IPv6 address. The `persistent-keepalive` of 25 seconds ensures regular handshakes, which is critical for the health monitoring system. A host route is added so the kernel knows to send traffic for the peer's mesh address through the WireGuard tunnel.

### Removing a peer

When a node leaves the mesh:

```sh
wg set nauka0 peer <pubkey> remove
ip -6 route del <mesh_ipv6>/128 dev nauka0
```

Route deletion errors are silently ignored since the route may have already been cleaned up by other means.

## Interface status

The module reads WireGuard status by parsing the output of `wg show nauka0 dump`, which provides a tab-separated machine-readable format. The status includes:

| Metric | Source |
|---|---|
| Listen port | Interface line, column 3 |
| Peer count | Number of peer lines |
| Total RX bytes | Sum of column 6 across all peers |
| Total TX bytes | Sum of column 7 across all peers |

This data is what you see when you run `nauka hypervisor status` -- it shows the tunnel's current health, traffic volume, and peer count.

## Per-peer handshake data

The health monitoring system uses per-peer handshake timestamps to determine reachability. These are also read from `wg show nauka0 dump`:

| Field | Description |
|---|---|
| Public key | The peer's WireGuard public key |
| Latest handshake | Unix timestamp of the last completed handshake (0 means never) |
| RX / TX bytes | Traffic counters for this specific peer |
| Endpoint | The peer's current endpoint as seen by WireGuard, or empty if unknown |

A peer with a `latest_handshake` of 0 has never successfully completed a WireGuard handshake. This is normal for newly added peers during their grace period (first 5 minutes).

## System requirements

The WireGuard interface module requires:

- **wireguard-tools** -- Provides the `wg` binary. Installed automatically during init/join.
- **iproute2** -- Provides the `ip` binary. Present on virtually all Linux systems.
- **WireGuard kernel module** -- Loaded automatically on modern Linux kernels (5.6+). On older kernels, the `wireguard` DKMS module must be installed separately.

If either the `wg` or `ip` binary is missing, init will fail with a clear error before any configuration is written.
