The fabric layer is transport-agnostic. All networking is abstracted behind a network backend interface, and you select the backend at init time with the `--mode` flag. This means the same orchestration logic works on the public internet, on a private datacenter network, and in test environments -- without any conditional logic in the higher layers.

## Available modes

### WireGuard (default)

The WireGuard backend creates an encrypted mesh using the WireGuard VPN protocol. Every node gets a private key, and all traffic between nodes travels through an encrypted tunnel on the `nauka0` interface. This is the right choice for any deployment where nodes communicate over the public internet or an untrusted network.

```sh
nauka hypervisor init --name node-1 --region eu --zone fsn1
nauka hypervisor init --name node-1 --region eu --zone fsn1 --mode wireguard
```

WireGuard is the default mode. You do not need to specify `--mode wireguard` explicitly.

The WireGuard backend manages:
- Installation of `wireguard-tools` via the system package manager
- Generation and management of `/etc/wireguard/nauka0.conf`
- The `nauka-wg` systemd service (wrapping `wg-quick`)
- Live configuration updates via `wg syncconf` (no tunnel teardown needed when peers change)

### Direct

The direct backend uses plain IP routing with no tunnel and no encryption. Nauka still manages the peer list and mesh IPv6 addressing, but it does not install WireGuard or create a tunnel interface. This is appropriate for clusters running entirely on a private datacenter network where the underlay is trusted and you want to avoid the overhead of encapsulation.

```sh
nauka hypervisor init --name node-1 --region eu --zone fsn1 --mode direct
```

Use direct mode only when you control the network infrastructure and trust the path between every node. Any network-level attacker between nodes can observe and modify mesh traffic in this mode.

### Mock

The mock backend is a no-op implementation used exclusively for testing. The interface is always reported as up, all operations succeed without touching the system, and no network traffic is sent. The mock backend is not useful for production deployments.

## Selecting a mode

Pass `--mode` to the `init` or `join` command:

```sh
nauka hypervisor init --name node-1 --mode wireguard
nauka hypervisor join --target 203.0.113.10 --name node-2 --mode direct
```

The mode is recorded in the fabric state and automatically used by all subsequent operations (`start`, `stop`, `leave`). Changing the mode after initialization requires a full `leave` and re-`init` cycle.

All nodes in a mesh should use the same mode. Mixing WireGuard and direct nodes is not supported -- they would not be able to communicate because WireGuard nodes expect encrypted tunnel traffic while direct nodes send unencapsulated packets.

## What the backend is responsible for

Every backend implements a common interface that the fabric operations layer calls without knowing which transport is in use:

| Capability | Description |
|---|---|
| Install | Install any required system packages (e.g., `wireguard-tools`) |
| Setup | Create the network interface, configure keys and addresses, add initial peers |
| Add/remove peer | Incrementally update the peer set during runtime |
| Update config | Reconcile the full peer list (used on restart or after joins) |
| Start / stop | Control the network service lifecycle |
| Teardown | Remove all configuration, uninstall the service |
| Status | Report interface state, listen port, peer count, traffic counters |

This design means adding a new transport backend in the future (for example, a VXLAN backend for overlay networking) requires implementing only this interface, with no changes to the fabric orchestration logic.
