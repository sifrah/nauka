The health monitoring system keeps the peer list accurate by continuously checking whether each peer is still reachable. It runs as a background task during peering mode and updates peer statuses based on WireGuard handshake activity.

## How health checks work

WireGuard performs a cryptographic handshake with each peer approximately every 2 minutes when there is active traffic. The health check system exploits this behavior: if a peer's last handshake is more than 5 minutes old, the tunnel is almost certainly down.

Each health check sweep:

1. Reads WireGuard handshake timestamps for all peers via `wg show nauka0 dump`
2. Compares each timestamp against the staleness threshold (300 seconds / 5 minutes)
3. Updates peer status to **Active** or **Unreachable** accordingly
4. Persists the updated state only if at least one status changed
5. Logs status transitions

## Timing parameters

| Parameter | Value | Rationale |
|---|---|---|
| Sweep interval | 30 seconds | Frequent enough to catch failures quickly, infrequent enough to avoid overhead |
| Staleness threshold | 300 seconds (5 minutes) | Deliberately longer than WireGuard's 2-minute handshake interval to avoid false positives during brief traffic gaps |
| New peer grace period | 300 seconds (5 minutes) | Newly added peers have not had time to establish a WireGuard handshake yet and should not be marked unreachable immediately |

The staleness threshold is set conservatively. A peer that has not completed a handshake in 5 minutes is very likely genuinely unreachable rather than merely experiencing a momentary gap in traffic. WireGuard's `PersistentKeepalive` (set to 25 seconds in Nauka) ensures regular handshakes even without application-level traffic.

## Grace period for new peers

When a peer is first added (e.g., it just joined the mesh), there is an inherent delay before WireGuard establishes the first handshake. Without special handling, the health check would immediately mark the new peer as unreachable, which would be misleading.

To handle this, the health check applies a grace period: any peer that was added within the last 5 minutes and has never completed a handshake is kept in **Active** status. Once the grace period expires, normal handshake-based status determination takes over.

## Database access pattern

The health check loop opens a fresh database connection at the start of each sweep and drops it when the sweep completes. This avoids holding a long-lived write lock that would block join requests, status queries, or other operations between sweeps.

Only sweeps that detect at least one status change trigger a database write. In a healthy mesh where all peers are active, the health check produces no writes and no log output -- silence means everything is working.

## Status transitions and logging

When a peer's status changes, the transition is logged:

```
INFO peer now active        peer=node-2
WARN peer now unreachable   peer=node-3
```

After any sweep that produces changes, a summary is logged:

```
INFO health check complete  total=3 active=2 unreachable=1 changed=1
```

Stable meshes with all peers active produce no log output at all. This makes it easy to spot problems in logs -- any health check output indicates a real status change.

## Integration with the mesh

The health check loop is one of several background tasks started by the peering command. It runs alongside the peering listener, announce listener, and reconciliation loop. The health check does not take corrective action itself -- it only updates peer status in the state database. Other subsystems use this status information:

- The announce broadcast skips peers marked as **Unreachable**, avoiding unnecessary connection timeouts
- The `nauka hypervisor status` command displays the current active and unreachable peer counts
- Future features may use status data for alerting or automatic remediation
