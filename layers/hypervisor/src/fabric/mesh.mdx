A Nauka mesh is a private network connecting a group of hypervisors. Every mesh is defined by two fundamental properties: a unique identifier and an IPv6 address space. These two values are shared by all members and cannot be changed after creation. Everything else -- the node's name, its WireGuard keys, its public endpoint -- is specific to each individual node.

## What defines a mesh

When you run `nauka hypervisor init`, two mesh-wide values are generated:

- **Mesh ID** -- A unique identifier (prefixed `mesh-`) that distinguishes this mesh from any other. Every node in the mesh shares the same mesh ID.
- **IPv6 prefix** -- A randomly generated `/48` ULA (Unique Local Address) prefix from the `fd00::/8` range. A `/48` prefix provides 2^80 possible addresses, far more than any practical cluster will ever require.

These values are transmitted to every node that joins the mesh through the peering protocol. They form the immutable foundation of the mesh's identity.

## How nodes get their addresses

Each node receives a `/128` IPv6 address derived deterministically from two inputs: the mesh's `/48` prefix and the node's WireGuard public key. Because WireGuard keys are cryptographically unique, every node is guaranteed a unique mesh address without any central coordination or DHCP.

This addressing scheme has several practical implications:

- Addresses are **stable and predictable** -- the same key always produces the same address.
- If a node leaves and rejoins with the same WireGuard key, it gets the same address. If it rejoins with a new key (which happens by default, since keys are regenerated on each init or join), it gets a new address.
- There is **no address allocation service** to manage or that could fail. Addresses are computed locally by each node.

## Node identity

Each hypervisor in the mesh carries a complete identity record:

| Property | Description |
|---|---|
| Hypervisor ID | Unique identifier for this node (prefixed `hv-`) |
| Name | Human-readable name, typically the hostname (e.g., `node-1`). Max 63 characters, lowercase alphanumeric and hyphens only |
| Region | Logical region label (e.g., `eu`, `us-east`) for topology awareness |
| Zone | Availability zone within the region (e.g., `fsn1`, `nbg1`) |
| WireGuard public key | Shared with all peers for tunnel establishment |
| WireGuard private key | Stored locally only, never transmitted over the network |
| WireGuard port | UDP listen port (default 51820) |
| Endpoint | Public IP:port that other nodes connect to over the internet |
| Mesh IPv6 | This node's unique `/128` address within the mesh prefix |

The private key is the most security-sensitive piece of the identity. It is persisted in the local state database but is never included in any network communication -- only the corresponding public key is shared with peers.

## Mesh secret and PIN

When a mesh is created, a mesh secret is generated (prefixed `syf_sk_`). This secret serves two purposes:

1. **PIN derivation** -- A short PIN is derived from the secret and used to authenticate join requests during peering. Without the correct PIN, a node cannot join the mesh.
2. **Membership credential** -- The secret is shared with every node that successfully joins, establishing shared trust across all members.

The mesh secret should be treated as a sensitive credential. Store it securely and share it only with operators who need to add nodes to the mesh. The PIN is displayed when `nauka hypervisor init` runs and is required by anyone running `nauka hypervisor join`.

## Validation rules

When creating a node identity (during both init and join), the following validation rules are enforced:

- **Name**: non-empty, max 63 characters, lowercase alphanumeric characters and hyphens only
- **Region**: same naming rules as name
- **Zone**: same naming rules as name
- **Port**: must be a valid non-zero port number

If validation fails, the operation aborts immediately -- no state is written and no network changes are made.

## Relationship to fabric state

Both the mesh identity and the node identity are stored as part of the fabric state, the complete snapshot of this node's mesh membership. You do not interact with these identities directly during normal operations. The `init` and `join` workflows handle all identity creation, and the `status` command displays the relevant information.
