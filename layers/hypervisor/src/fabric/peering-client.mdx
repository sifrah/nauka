The peering client is the component that initiates a join. When you run `nauka hypervisor join`, the peering client connects to the target node's peering port, authenticates with the PIN, and retrieves everything needed to become a mesh member.

You do not interact with the peering client directly -- it is invoked automatically by the join operation. This page explains its behavior so you can understand and troubleshoot the join process.

## What happens during a join connection

1. **Address resolution** -- The target address from `--target` is parsed. If no port is specified, the default peering port (51821) is used. Both `IP:port` and bare `IP` formats are accepted.
2. **TCP connection** -- A TCP connection is established to the target's peering port.
3. **TLS handshake** -- A TLS session is established using the TOFU (Trust on First Use) model. The client accepts any server certificate without verification. Authentication is handled by the PIN, not by the certificate.
4. **Send join request** -- The client sends a length-prefixed JSON message containing the node's name, region, zone, WireGuard public key, listen port, endpoint (if provided), and PIN.
5. **Receive join response** -- The client reads the response from the server.
6. **Check acceptance** -- If the response indicates rejection, the client returns an error with the rejection reason and the join aborts.
7. **Return response** -- On success, the response (containing the mesh secret, prefix, mesh ID, and peer list) is returned to the operations layer, which completes the local setup.

## Prerequisites for joining

Before running the join command, ensure:

| Requirement | Details |
|---|---|
| Target address | The public IP or hostname of any running mesh member |
| PIN | Displayed when the mesh was created with `nauka hypervisor init` |
| Peering port open | The target node must have WireGuard port + 1 (default 51821) reachable over TCP |
| Peering mode active | The target node must be running `nauka hypervisor peering` |

## Troubleshooting join failures

**Connection refused** -- The target node's peering port is not reachable. Verify that the target is running peering mode (`nauka hypervisor peering` on that node) and that the firewall allows inbound TCP on the peering port (default 51821).

**TLS handshake failed** -- This usually indicates a network-level issue (connection reset, timeout) rather than a certificate problem, since the client accepts any certificate. Check network connectivity between the two nodes.

**Invalid PIN** -- The PIN does not match what the target expects. The PIN is displayed once when `nauka hypervisor init` runs. Verify you have the correct PIN for the target mesh.

**IP blocked** -- If you have entered the wrong PIN 5 or more times from the same IP address, the peering server blocks that IP for 10 minutes. Wait for the block to expire or correct the PIN.

**Name conflict** -- The name specified with `--name` is already in use by another node in the mesh. Choose a different name and retry.

**Already initialized** -- The node already has fabric state from a previous init or join. Run `nauka hypervisor leave` first to clean up, then retry the join.

## Retry behavior

The peering client does not implement automatic retries. If the connection fails, the error is returned directly to the operator. This is by design -- join failures typically indicate a configuration or network problem that requires human attention (wrong PIN, firewall blocking the port, target not running peering mode).

To retry a failed join, simply run the `nauka hypervisor join` command again.
