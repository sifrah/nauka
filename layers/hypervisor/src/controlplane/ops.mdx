The operations module orchestrates the control plane lifecycle. It sits above the [service module](/nauka/layers/hypervisor/controlplane/service) and combines binary installation, configuration, systemd management, and health checks into three high-level operations: bootstrap, join, and leave.

## Bootstrap (first node)

When you run `nauka hv init`, the control plane bootstrap creates a single-node TiKV cluster on the init node:

<Mermaid chart={`
sequenceDiagram
  participant Init as nauka hv init
  participant Svc as Service layer
  participant PD
  participant TiKV

  Init->>Svc: ensure_installed()
  Note over Svc: Download TiUP, install PD + TiKV

  Init->>Svc: install(pd_cfg, tikv_cfg)
  Note over Svc: Write configs + systemd units

  Init->>Svc: enable_and_start()
  Svc->>PD: systemctl start nauka-pd
  Svc->>TiKV: systemctl start nauka-tikv

  Init->>PD: wait_pd_ready (30s timeout)
  Init->>TiKV: wait_tikv_ready (60s timeout)
`} />

The PD configuration is generated with `initial-cluster-state = "new"`, which tells PD to create a fresh Raft group with itself as the only member. TiKV is configured to point at the local PD endpoint.

After both services pass their health checks, the control plane is ready to accept joins from other nodes.

## Join (subsequent nodes)

When a node runs `nauka hv join`, the control plane join decides whether to run PD based on how many peers are already in the cluster:

| Peers at join time | Total nodes | This node runs |
|---|---|---|
| 1 peer | 2nd node | PD + TiKV |
| 2 peers | 3rd node | PD + TiKV |
| 3+ peers | 4th+ node | TiKV only |

The maximum number of PD members is **3**. Once three PD members exist, additional nodes run TiKV only.

### Joining with PD (nodes 2 and 3)

1. **Verify mesh connectivity** -- Poll the existing PD's health endpoint (`/pd/api/v1/health`) with a 30-second timeout to confirm the mesh is working
2. **Clean up zombie members** -- Remove any unhealthy PD members with no name (artifacts from failed previous joins)
3. **Install PD in join mode** -- The `initial-cluster` field is omitted from `pd.toml`; instead, the systemd unit's `ExecStart` uses `--join=<primary_pd_url>` to register with the existing cluster
4. **Install TiKV** -- TiKV config includes all known PD endpoints (existing peers plus this node)
5. **Start and wait** -- PD starts first, then TiKV; health checks allow up to 120 seconds each

### Joining TiKV only (nodes 4+)

1. **Verify mesh connectivity** -- Same health check against an existing PD
2. **Install TiKV only** -- No PD unit is created; TiKV config points to the existing PD cluster
3. **Start TiKV** -- `systemctl enable --now nauka-tikv`; waits up to 60 seconds for the store to register with PD

## Leave

When a node runs `nauka hv leave`, the control plane is torn down in a controlled sequence:

1. **Deregister TiKV store** -- Query `GET /pd/api/v1/stores` to find this node's store ID (matched by mesh IPv6 address), then `DELETE /pd/api/v1/store/{id}` to remove it. This prevents PD from waiting indefinitely for a store that will never return.
2. **Deregister PD member** -- If this node runs PD, query `GET /pd/api/v1/members` to find the member ID (matched by peer URL containing the mesh IPv6), then `DELETE /pd/api/v1/members/id/{id}`.
3. **Uninstall** -- Stop and disable both services, remove systemd units, config files, and data directories, then reload the systemd daemon.

<Aside type="note">
If the mesh IPv6 address is not available (corrupted state), the leave falls back to a simple uninstall without deregistration. The cluster will eventually detect the missing member, but this is less clean than a proper deregistration.
</Aside>

## Status

The `status` function queries the local systemd service state and the PD HTTP API to return:

- Whether `nauka-pd` and `nauka-tikv` are active
- Number of PD members in the cluster
- Number of TiKV stores registered
- Name of the current PD leader

This information is used by both `nauka hv status` and `nauka hv doctor`.

## Operational commands

```sh
# Start the control plane (requires prior init or join)
nauka hv start

# Stop the control plane
nauka hv stop

# Check control plane health
nauka hv doctor
```

Control plane start and stop are handled automatically as part of the hypervisor-wide `start` and `stop` commands. There are no separate CLI commands for the control plane alone.
