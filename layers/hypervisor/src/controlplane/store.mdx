The ClusterDb is the interface that Nauka uses to store and retrieve cluster-wide state in TiKV. It is used by storage operations to publish and fetch region configurations, and will be used by future sub-systems (VMs, networks, users) to manage their distributed state.

## How it works

ClusterDb connects to the TiKV cluster through one or more PD endpoints. The TiKV client library handles shard discovery and request routing automatically -- you connect to PD, and PD directs you to the right TiKV node for each key.

## Key structure

All keys are partitioned by namespace using the format `{namespace}/{key}`. This prevents collisions between different sub-systems that share the same TiKV cluster.

For example, storage region configs are stored under the namespace `storage/regions`:
- `storage/regions/eu` -- configuration for the "eu" region
- `storage/regions/us-east` -- configuration for the "us-east" region

## Operations

ClusterDb provides six operations for working with distributed state:

| Operation | Description |
|---|---|
| **put** | Write a value (JSON-serialized) |
| **get** | Read a value (JSON-deserialized); returns nothing if the key does not exist |
| **delete** | Remove a key |
| **list** | Scan all keys matching a prefix within a namespace (up to 10,000 results) |
| **exists** | Check if a key exists without reading its value |
| **batch_put** | Write multiple key-value pairs in a single call (not transactional, but more efficient than individual puts) |

All values are stored as JSON. Serialization and deserialization happen automatically.

## Current usage

Today, ClusterDb is used for one purpose: distributing storage region configurations across the cluster.

When the first node runs `nauka hv init` with S3 credentials, the region configuration (endpoint, bucket, access key, secret key, encryption password) is published to TiKV under `storage/regions/{region}`.

When subsequent nodes run `nauka hv join`, they fetch the region configuration from TiKV and set up their local ZeroFS instance with the same S3 backend. This means S3 credentials only need to be provided once during init -- all joining nodes receive them automatically through the control plane.

<Aside type="note">
The fetch operation retries up to 6 times with a 2-second delay between attempts. This handles transient "Leader not found" errors that can occur immediately after a node joins the cluster, while Raft is still electing a leader.
</Aside>

## Relationship to local state

Each Nauka node also has a local SurrealDB instance (`EmbeddedDb`) backed by the on-disk SurrealKV engine for node-specific state like the mesh identity and peer list. The two stores serve different purposes:

| | EmbeddedDb | ClusterDb |
|---|---|---|
| **Scope** | Single node | Entire cluster |
| **Backend** | Embedded SurrealDB (SurrealKV) | Distributed (TiKV) |
| **Survives node loss** | No | Yes (replicated) |
| **Requires mesh** | No | Yes (PD + TiKV on mesh IPv6) |
| **Use case** | Node identity, peer list, region registry | Shared config, VM records, network allocations |

Both stores expose SurrealQL-flavoured semantics, so patterns developed against one transfer naturally to the other when Phase 2 swaps the distributed backend to `EmbeddedDb<TiKv>`.
