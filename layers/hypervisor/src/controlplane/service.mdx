The service module handles the low-level mechanics of running PD and TiKV on a node: binary installation via TiUP, configuration file generation, systemd unit management, health polling, and cluster membership operations.

## TiUP installation

TiUP is the package manager for PD and TiKV. Nauka installs it automatically the first time a node initializes or joins a cluster.

The installation process:

1. Download the TiUP binary for the current architecture (`amd64` or `arm64`) from the PingCAP mirror
2. Extract it to `/opt/nauka/tiup/bin/tiup`
3. Configure the TiUP mirror URL
4. Install the `pd` and `tikv` components via `tiup install pd tikv`

If TiUP and both components are already present, the call is a no-op.

<Aside type="note">
The `TIUP_HOME` environment variable is set to `/opt/nauka/tiup` for all TiUP and service operations. This keeps TiUP isolated from any user-level installation.
</Aside>

## PD configuration

The PD config file is written to `/etc/nauka/pd.toml`. Key settings:

| Setting | Value |
|---|---|
| Client URL | `http://[mesh_ipv6]:2379` |
| Peer URL | `http://[mesh_ipv6]:2380` |
| Advertise URLs | Same as client and peer URLs |
| Data directory | `/var/lib/nauka/pd` |
| Log file | `/var/log/nauka/pd.log` |
| Log max size | 50 MB (auto-rotation) |
| Log level | `warn` |

For the bootstrap node, the config includes:
- `initial-cluster = "node-name=http://[mesh_ipv6]:2380"`
- `initial-cluster-state = "new"`

For joining nodes, the `initial-cluster` fields are omitted entirely. Instead, the systemd unit's `ExecStart` line uses the `--join=<primary_pd_url>` flag to register with the existing cluster.

## TiKV configuration

The TiKV config file is written to `/etc/nauka/tikv.toml`. Key settings:

| Setting | Value |
|---|---|
| Server address | `[mesh_ipv6]:20160` |
| Advertise address | Same as server address |
| PD endpoints | List of all known PD client URLs |
| Data directory | `/var/lib/nauka/tikv` |
| Log file | `/var/log/nauka/tikv.log` |
| Log max size | 50 MB (auto-rotation) |
| Log level | `warn` |

## Systemd units

### nauka-pd.service

```ini
[Unit]
Description=Nauka Placement Driver (PD)
After=network-online.target nauka-wg.service
Wants=network-online.target
Requires=nauka-wg.service

[Service]
Type=simple
Environment=HOME=/root
Environment=TIUP_HOME=/opt/nauka/tiup
ExecStart=/opt/nauka/tiup/bin/tiup pd --config=/etc/nauka/pd.toml
Restart=on-failure
RestartSec=5
LimitNOFILE=1000000

[Install]
WantedBy=multi-user.target
```

For joining nodes, the `ExecStart` line appends `--join=<primary_pd_url>`.

### nauka-tikv.service

```ini
[Unit]
Description=Nauka TiKV Storage Engine
After=network-online.target nauka-wg.service
Wants=network-online.target
Requires=nauka-wg.service

[Service]
Type=simple
Environment=HOME=/root
Environment=TIUP_HOME=/opt/nauka/tiup
ExecStart=/opt/nauka/tiup/bin/tiup tikv --config=/etc/nauka/tikv.toml
Restart=on-failure
RestartSec=5
LimitNOFILE=1000000

[Install]
WantedBy=multi-user.target
```

Both units require `nauka-wg.service` because PD and TiKV communicate over mesh IPv6 addresses. The WireGuard mesh must be up before either service can start.

## Startup and shutdown order

Order matters. PD must be running before TiKV starts, because TiKV registers itself with PD on startup.

| Operation | Sequence |
|---|---|
| `enable_and_start` | Start PD, wait 3 seconds, then start TiKV |
| `start` | Start PD, wait 2 seconds, then start TiKV |
| `stop` | Stop TiKV first, then stop PD |
| `restart` | Stop both, wait 1 second, then start in order |
| `reload` | Rewrite config files; PD gets `reload-or-restart`, TiKV gets a full restart |

## Health polling

Two readiness checks block until the service is healthy or a timeout is reached:

**PD readiness** -- Polls `GET http://[mesh_ipv6]:2379/pd/api/v1/health` every 1 second. Returns success when a 200 response is received.

**TiKV readiness** -- Polls `GET http://[mesh_ipv6]:2379/pd/api/v1/stores` every 1 second. Returns success when the response contains at least one registered store (`"count"` is non-zero).

| Context | PD timeout | TiKV timeout |
|---|---|---|
| Bootstrap (init) | 30 seconds | 60 seconds |
| Join with PD | 120 seconds | 120 seconds |
| Join TiKV only | -- | 60 seconds |

Joining nodes get longer timeouts because they must first establish mesh connectivity to an existing PD member.

## Cluster status

The `cluster_status` function queries the PD HTTP API and returns:

- `pd_active` -- whether the local PD service is running
- `tikv_active` -- whether the local TiKV service is running
- `pd_members` -- total number of PD members in the cluster
- `tikv_stores` -- total number of TiKV stores registered with PD
- `leader` -- name of the current PD leader (if elected)

## Uninstall

The `uninstall` function:

1. Disables and stops both `nauka-tikv` and `nauka-pd`
2. Removes systemd unit files from `/etc/systemd/system/`
3. Removes config files from `/etc/nauka/`
4. Deletes data directories (`/var/lib/nauka/pd/` and `/var/lib/nauka/tikv/`)
5. Runs `systemctl daemon-reload`
