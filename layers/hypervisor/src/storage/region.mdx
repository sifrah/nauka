A region is the unit of storage configuration in Nauka. Each region maps to exactly one S3 bucket on one endpoint. The region module defines the configuration model and the local registry that persists configured regions on each node.

## Region configuration

Each region requires the following settings:

| Field | Required | Description |
|---|---|---|
| `region` | Yes | Short identifier (e.g., `eu`, `us-east`). Used in service names, file paths, and CLI output. |
| `s3_endpoint` | Yes | S3-compatible endpoint URL (must start with `http://` or `https://`) |
| `s3_bucket` | Yes | Bucket name |
| `s3_access_key` | Yes | S3 access key |
| `s3_secret_key` | Yes | S3 secret key |
| `s3_region` | No | AWS region string (e.g., `eu-central-1`). Leave empty for non-AWS providers. |
| `encryption_password` | Auto | Derived deterministically from `SHA256(s3_secret_key + ":zerofs:" + region)` |
| `is_default` | No | Whether this is the default region when none is specified |

### Validation

Before a region is saved or used, all required fields are validated:

- `region` must be non-empty
- `s3_endpoint` must be non-empty and start with `http://` or `https://`
- `s3_bucket` must be non-empty
- `s3_access_key` must be non-empty
- `s3_secret_key` must be non-empty

If any validation fails, the setup is rejected with a descriptive error before any files are written or services started.

## Region registry

The region registry is the local collection of all configured storage regions on a node. It is persisted to the node's local state store under the path `regions/config`.

### How it works

- **Add or update** -- When a region is set up, its configuration is added to the registry. If a region with the same identifier already exists, it is replaced.
- **Remove** -- When a region is removed, its entry is deleted from the registry.
- **Find** -- Look up a specific region by its identifier.
- **Default region** -- Returns the region marked as `is_default`, or the first region in the list if none is explicitly marked as default.

### Persistence

The registry is loaded at the start of any storage operation and saved after any mutation. Saves are atomic -- a write either fully commits or leaves the previous value intact.

<Aside type="note">
The registry is local to each node. There is no automatic replication of the registry across nodes. However, the underlying region configuration (S3 endpoint, bucket, credentials) is distributed through TiKV during cluster join, so all nodes in the same region end up with the same S3 backend configuration.
</Aside>

## Configuration flow

During cluster lifecycle, region configuration flows as follows:

1. **Init node** -- The operator provides S3 credentials via `--s3-*` flags. The encryption password is derived. The configuration is saved locally and published to TiKV.
2. **Joining node** -- The node fetches the region configuration from TiKV (no S3 credentials needed on the command line). The configuration is saved to the local registry and used to install ZeroFS.
3. **All nodes** -- The local registry is the authoritative source for which regions this specific node serves. ZeroFS configs and systemd units are generated from the registry entries.

## Where credentials are stored

Region credentials (S3 access key and secret key) are stored in two places:

- **Local registry** -- Part of the node's local state file
- **ZeroFS TOML config** -- Written directly into `/etc/nauka/zerofs/{region}.toml` (file permissions `0600`)

The encryption password is also written to the ZeroFS config, since ZeroFS needs it to encrypt and decrypt data at rest.
