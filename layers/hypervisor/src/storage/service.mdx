The service module handles the ZeroFS binary lifecycle and per-region systemd services. Each configured region gets its own isolated ZeroFS instance managed by a dedicated service unit.

## Binary installation

ZeroFS is installed automatically when a storage region is first set up on a node. The binary is placed at `/usr/local/bin/zerofs`.

The installation downloads a pre-compiled, PGO-optimized release from GitHub:

```
https://github.com/barre/zerofs/releases/latest/download/zerofs-pgo-multiplatform.tar.gz
```

The correct binary is extracted based on the server's architecture:

| Architecture | Binary name |
|---|---|
| x86_64 | `zerofs-linux-amd64-pgo` |
| aarch64 | `zerofs-linux-arm64-pgo` |

If ZeroFS is already installed, the installation step is skipped.

## Per-region configuration

Each region gets a TOML config file at `/etc/nauka/zerofs/{region}.toml`. The file is written with `0600` permissions (owner-readable only) because it contains S3 credentials.

The generated configuration includes:

**Cache settings:**
- Disk cache directory: `/var/lib/nauka/zerofs/{region}/`
- Disk cache size: 10 GB
- Memory cache size: 1 GB

**Storage backend:**
- S3 URL: `s3://{bucket}/zerofs-{region}`
- Encryption password (derived from S3 secret key + region)
- S3 access key and secret key written directly into the config

**Servers:**
- NBD: `127.0.0.1:10809` with Unix socket at `/tmp/zerofs-{region}.nbd.sock`
- RPC: `127.0.0.1:7000` with Unix socket at `/tmp/zerofs-{region}.rpc.sock`

**Other:**
- Compression: LZ4
- Telemetry: disabled

<Aside type="note">
S3 credentials are written directly into the TOML config file (not via environment variables). The file's `0600` permissions ensure only root can read them.
</Aside>

For S3 endpoints using plain HTTP (non-TLS), the config automatically includes `allow_http = "true"`.

## Systemd unit

Each region gets a systemd unit at `/etc/systemd/system/nauka-zerofs-{region}.service`:

```ini
[Unit]
Description=Nauka ZeroFS Storage ({region})
After=network-online.target nauka-wg.service
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/local/bin/zerofs run -c /etc/nauka/zerofs/{region}.toml
Restart=on-failure
RestartSec=5
LimitNOFILE=1000000

[Install]
WantedBy=multi-user.target
```

The unit depends on `nauka-wg.service` (the WireGuard mesh) and `network-online.target`. It restarts automatically on failure with a 5-second backoff.

## Service lifecycle

| Operation | Command | Effect |
|---|---|---|
| Install | `install_region(config)` | Create directories, write TOML config (0600), write systemd unit, reload daemon |
| Start | `start_region(region)` | `systemctl enable --now nauka-zerofs-{region}` |
| Stop | `stop_region(region)` | `systemctl stop nauka-zerofs-{region}` |
| Check status | `is_region_active(region)` | `systemctl is-active --quiet nauka-zerofs-{region}` |
| Uninstall one | `uninstall_region(region)` | Stop and disable service, remove unit + config + cache directory |
| Uninstall all | `uninstall_all()` | Find all `nauka-zerofs-*` units and remove them, plus all config, cache, and mount directories |

## Directories

| Path | Purpose | Created by |
|---|---|---|
| `/etc/nauka/zerofs/` | Per-region TOML config files | `install_region` |
| `/var/lib/nauka/zerofs/{region}/` | Local disk cache (up to 10 GB per region) | `install_region` |
| `/mnt/nauka/{region}/` | Mount point for ZeroFS | `install_region` |

## Service naming

Service names follow the pattern `nauka-zerofs-{region}`, where `{region}` is the region identifier provided during setup. Examples:

- `nauka-zerofs-eu`
- `nauka-zerofs-us-east`
- `nauka-zerofs-default`

Use standard systemd commands to inspect these services:

```sh
systemctl status nauka-zerofs-eu
journalctl -u nauka-zerofs-eu -f
```
