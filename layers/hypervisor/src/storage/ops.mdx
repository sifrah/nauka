The storage operations module is the top-level entry point for storage lifecycle on a node. It coordinates config validation, ZeroFS installation, systemd service management, registry persistence, and configuration distribution through the control plane.

## Setting up a region

When a node initializes or joins a cluster, storage is set up automatically for the node's region. The setup runs four steps:

<Mermaid chart={`
sequenceDiagram
  participant H as Handler
  participant Ops as Storage ops
  participant Svc as Service
  participant Reg as Registry

  H->>Ops: setup_region(config)
  Ops->>Ops: validate config
  Ops->>Svc: ensure_installed()
  Note over Svc: Download ZeroFS if missing
  Ops->>Svc: install_region(config)
  Note over Svc: Write TOML + systemd unit
  Ops->>Svc: start_region(region)
  Note over Svc: systemctl enable --now
  Ops->>Reg: save to local registry
`} />

1. **Validate** -- Checks that the region name, S3 endpoint, bucket, and credentials are all provided and that the endpoint URL starts with `http://` or `https://`
2. **Install ZeroFS** -- Downloads the ZeroFS binary if not already present
3. **Install config and unit** -- Writes the region-specific TOML config and systemd service unit
4. **Start and persist** -- Starts the service and saves the region configuration to the local registry

## Removing a region

Stops the region's systemd service, removes its config file and unit, deletes the local cache directory, and removes the entry from the local registry. Safe to call on a region that is already stopped.

## Starting and stopping all regions

The `start` and `stop` commands on the hypervisor automatically manage all configured storage regions:

- **Start all** -- Iterates the local registry and starts any region whose service is not currently active. Already-running regions are skipped.
- **Stop all** -- Stops every registered region regardless of current state.

Both operations handle per-region errors gracefully -- one failing region does not prevent the others from being started or stopped.

## Leaving the cluster

When a node runs `nauka hv leave`, storage cleanup removes all ZeroFS instances:

1. Discovers all `nauka-zerofs-*` systemd units
2. Disables and stops each service
3. Removes all config files from `/etc/nauka/zerofs/`
4. Deletes all cache directories under `/var/lib/nauka/zerofs/`
5. Removes mount points under `/mnt/nauka/`

This runs before the control plane teardown, since ZeroFS may depend on mesh connectivity.

## Configuration distribution via TiKV

Region storage configurations are shared across the cluster through the distributed control plane (TiKV), stored under the namespace `storage/regions`.

### Publishing (during init)

When the first node initializes with S3 credentials, the complete region configuration -- including endpoint, bucket, credentials, and derived encryption password -- is published to TiKV. This happens automatically after the control plane is bootstrapped.

### Fetching (during join)

When a node joins the cluster, it fetches the region configuration from TiKV before setting up local storage. This means S3 credentials only need to be provided once during init.

<Aside type="note">
The fetch operation retries up to 6 times with a 2-second delay between attempts to handle transient TiKV errors (e.g., "Leader not found") that occur while Raft is still electing a leader on a newly joined node.
</Aside>

## Checking status

The status function returns a snapshot of all registered regions on the current node:

| Field | Description |
|---|---|
| region | Region identifier |
| s3_endpoint | S3 endpoint URL |
| s3_bucket | Bucket name |
| active | Whether the systemd service is currently running |
| is_default | Whether this is the node's default storage region |

If the registry cannot be loaded (e.g., during degraded startup), an empty list is returned rather than an error, so status checks remain non-fatal.
