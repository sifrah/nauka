The hypervisor handler layer defines the complete CLI surface for managing Nauka nodes. It declares every command and its arguments through a resource definition, then routes each command to the appropriate sub-system.

## Command routing

When you run a `nauka hypervisor` command, the request flows through three layers:

<Mermaid chart={`
graph LR
  CLI[CLI Parser] --> H[Handler]
  H --> F[Fabric ops]
  H --> C[Controlplane ops]
  H --> S[Storage ops]
  H --> D[Doctor]
`} />

The handler extracts arguments from the parsed request and delegates to the correct sub-system. It does not contain business logic -- all operational work happens in the ops modules.

## Full command reference

### init

Create a new cluster and initialize this server as the first node.

| Argument | Required | Default | Description |
|---|---|---|---|
| `--name` | No | hostname | Node name |
| `--region` | No | `default` | Region label |
| `--zone` | No | `default` | Zone label |
| `--port` | No | `51820` | WireGuard listen port (also uses port+1 for peering, port+2 for announce) |
| `--mode` | No | `wireguard` | Network mode: `wireguard`, `direct`, or `mock` |
| `--interface` | No | auto-detect | Fabric network interface (e.g., `eth1`) |
| `--endpoint` | No | auto-detect | Public endpoint IP for peering |
| `--s3-endpoint` | Yes | -- | S3 endpoint URL for region storage |
| `--s3-bucket` | Yes | -- | S3 bucket name |
| `--s3-access-key` | Yes | -- | S3 access key |
| `--s3-secret-key` | Yes | -- | S3 secret key |
| `--s3-region` | No | empty | S3 region (e.g., `eu-central-1`) |
| `--peering` | No | false | Start peering listener after init |

```sh
nauka hv init --name my-cloud --region eu --zone fsn1 \
  --s3-endpoint https://s3.eu.example.com \
  --s3-bucket nauka-eu \
  --s3-access-key AKID \
  --s3-secret-key SECRET \
  --peering
```

The init process runs 8 steps in WireGuard mode: 2 for fabric setup, 4 for control plane bootstrap, and 2 for storage configuration.

### join

Add this server to an existing cluster.

| Argument | Required | Default | Description |
|---|---|---|---|
| `--target` | Yes | -- | IP or IP:port of an existing node |
| `--name` | No | hostname | Node name |
| `--pin` | No | -- | PIN for authentication |
| `--region` | No | `default` | Region label |
| `--zone` | No | `default` | Zone label |
| `--port` | No | `51820` | WireGuard listen port |
| `--mode` | No | `wireguard` | Network mode |

```sh
nauka hv join --target 46.224.166.60 --pin G7CCZX --region eu --zone nbg1
```

The join process runs 7 steps in WireGuard mode: 2 for fabric, 3 for control plane, 1 for storage setup, and 1 for peer announcement.

<Aside type="note">
Storage credentials are not needed during join. The joining node fetches the region's S3 configuration from the distributed control plane (TiKV), which was published by the init node.
</Aside>

### status

Show the current state of this node, including mesh identity, service health, peer count, and control plane membership.

```sh
nauka hv status
```

Returns: node name, ID, region, zone, mesh IPv6 address, overall state (`available`, `degraded`, or `down`), WireGuard status, peer count, PD and TiKV status, PD member and store counts, and the current PD leader.

The node state is determined as:
- **available** -- both WireGuard and TiKV are running
- **degraded** -- WireGuard is running but TiKV is not
- **down** -- WireGuard is not running

### start

Start all hypervisor services in dependency order: fabric, then announce listener, then control plane, then storage.

```sh
nauka hv start
```

### stop

Stop all services in reverse dependency order: storage, then control plane, then announce listener, then fabric.

```sh
nauka hv stop
```

### leave

Gracefully remove this node from the cluster. Requires confirmation.

```sh
nauka hv leave
```

The leave process runs 4 steps:
1. Stop all storage services (ZeroFS instances)
2. Deregister the TiKV store and PD member from the cluster, then uninstall control plane services
3. Notify peers and tear down the mesh
4. Remove all services and configuration files

### peering

Start the peering listener to accept new nodes joining the cluster. The listener runs on the WireGuard port + 1 and displays the PIN needed for authentication.

| Argument | Required | Default | Description |
|---|---|---|---|
| `--timeout` | No | `3600` | Listener timeout in seconds |

```sh
nauka hv peering
```

### list

List all known hypervisors in the cluster (this node plus all peers).

```sh
nauka hv list
```

Output columns: NAME, REGION, ZONE, STATE, CPU, MEMORY, VMs.

### get

Show detailed information about a specific hypervisor by name.

```sh
nauka hv get web-01
```

Output fields: Name, ID, Region, Zone, Address (mesh IPv6), State.

### doctor

Run health diagnostics across all sub-systems. See [Doctor](/nauka/layers/hypervisor/doctor) for full details.

```sh
nauka hv doctor
```

### announce-listen

Internal command that runs the persistent announce listener. Typically managed as a systemd service rather than run directly.

| Argument | Required | Default | Description |
|---|---|---|---|
| `--port` | No | `51822` | Announce listen port |

### drain / enable

Reserved for future use. `drain` will evacuate all VMs before maintenance. `enable` will re-enable a node for VM scheduling.
