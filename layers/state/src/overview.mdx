The state layer handles all data persistence in Nauka. It provides two storage backends that serve different purposes at different stages of a node's lifecycle.

## Two-tier storage

Nauka uses a two-tier approach to persistence because a distributed database is not available when a node first boots. The two tiers are:

- **Local store (LocalDb)** — a JSON file on each node, used for bootstrap state that must exist before the cluster database is reachable.
- **Cluster store (ClusterDb)** — a TiKV-backed distributed key-value store, replicated across the mesh via Raft. Used for all shared state once the cluster is running.

Platform engineers do not choose which tier to use at runtime. Each piece of state has a fixed home: bootstrap data lives locally, everything else lives in the cluster store.

## Local store

The local store persists the minimum state a node needs to function independently: its mesh identity, WireGuard keys, and peer list. This data must be readable even when the network is down or the cluster database has not yet been deployed.

### File location

Each layer gets its own JSON file at:

```
~/.nauka/{layer}.json
```

For example, the hypervisor layer stores its state at `~/.nauka/fabric.json`. If you need to inspect or back up a node's bootstrap state, this is where to look.

### File permissions

The `~/.nauka/` directory is created with mode `0700` and each JSON file with mode `0600`. Only the owning user can read or write these files. This matters because the files contain WireGuard private keys and mesh credentials.

### Data format

The JSON file is a flat key-value map. Keys are structured as `{table}/{key}`, where `table` groups related records and `key` identifies a specific entry. Values are JSON objects. The file is human-readable and can be inspected with any JSON tool.

### Persistence guarantees

Every write flushes the entire file to disk immediately. There is no write-ahead log or batching. This is acceptable because the local store holds a small amount of slowly-changing data (typically a few kilobytes). If the process crashes mid-write, the file may be truncated; on next open, a parse failure will surface as an error.

### When the local store is used

The local store is the only backend involved during these operations:

- `nauka hypervisor init` — creates mesh identity and WireGuard keys
- `nauka hypervisor join` — stores the received mesh credentials and peer information
- `nauka hypervisor start` / `stop` — reads local state to configure the WireGuard interface
- `nauka hypervisor leave` — clears local mesh state

In all of these cases, the node may not yet have connectivity to the cluster database.

## Cluster store

Once the mesh is established and TiKV is running, all shared state moves to the cluster store. This includes resources like virtual machines, VPCs, networks, subnets, users, and volumes.

The cluster store is covered in detail on the [Cluster Store](/layers/state/cluster/) page.

## Separation of concerns

A node's local JSON file and the cluster database never store the same data. The local store holds per-node identity; the cluster store holds shared resources. This means there is no synchronization or conflict resolution between the two tiers. Each tier is authoritative for its own data.

If you need to reset a single node without affecting the cluster, delete its `~/.nauka/*.json` files and re-join the mesh. If you need to reset the entire cluster's state, that is a TiKV operation and does not involve the local files.
