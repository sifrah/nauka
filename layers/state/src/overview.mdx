The state layer handles all data persistence in Nauka. It provides two storage backends that serve different purposes at different stages of a node's lifecycle.

## Two-tier storage

Nauka uses a two-tier approach to persistence because a distributed database is not available when a node first boots. The two tiers are:

- **Local store (`EmbeddedDb::open`)** — an embedded SurrealDB instance on each node, backed by the SurrealKV on-disk engine. Used for bootstrap state that must exist before the cluster database is reachable.
- **Cluster store (`EmbeddedDb::open_tikv`)** — the same SurrealDB API, this time against a TiKv cluster replicated across the mesh via Raft. Used for all shared state once the cluster is running.

Platform engineers do not choose which tier to use at runtime. Each piece of state has a fixed home: bootstrap data lives locally, everything else lives in the cluster store.

## Local store

The local store persists the minimum state a node needs to function independently: its mesh identity, WireGuard keys, and peer list. This data must be readable even when the network is down or the cluster database has not yet been deployed.

### File location

Each node stores its bootstrap state in a single SurrealKV datastore at:

```
~/.nauka/bootstrap.skv
```

(or `/var/lib/nauka/bootstrap.skv` in service mode, as resolved by `nauka_core::process::nauka_db_path`). If you need to inspect or back up a node's bootstrap state, this is where to look.

### File permissions

The `~/.nauka/` directory is created with mode `0700`, and the SurrealKV datastore directory inherits a matching set of protective permissions. Only the owning user can read or write these files. This matters because they contain WireGuard private keys and mesh credentials.

### Data model

The bootstrap schema (`schemas/bootstrap.surql`) defines the SCHEMAFULL tables for mesh, hypervisor, peer, and WireGuard key records. `EmbeddedDb::open` applies this schema idempotently at every startup via `DEFINE ... IF NOT EXISTS`, so adding new fields or tables in later releases remains forward-compatible.

### Persistence guarantees

SurrealKV is an LSM-backed engine with a write-ahead log. Writes are durable once the underlying commit returns. `EmbeddedDb::shutdown` polls the OS-level flock on the datastore so that by the time it returns, every committed write is durably on disk and the file can be safely reopened by the next caller.

### When the local store is used

The local store is the only backend involved during these operations:

- `nauka hypervisor init` — creates mesh identity and WireGuard keys
- `nauka hypervisor join` — stores the received mesh credentials and peer information
- `nauka hypervisor start` / `stop` — reads local state to configure the WireGuard interface
- `nauka hypervisor leave` — clears local mesh state

In all of these cases, the node may not yet have connectivity to the cluster database.

## Cluster store

Once the mesh is established and TiKV is running, all shared state moves to the cluster store. This includes resources like virtual machines, VPCs, networks, subnets, users, and volumes.

The cluster store is covered in detail on the [Cluster Store](/layers/state/cluster/) page.

## Separation of concerns

A node's local SurrealKV datastore and the cluster database never store the same data. The local store holds per-node identity; the cluster store holds shared resources. This means there is no synchronization or conflict resolution between the two tiers. Each tier is authoritative for its own data.

If you need to reset a single node without affecting the cluster, delete its `~/.nauka/bootstrap.skv/` directory and re-join the mesh. If you need to reset the entire cluster's state, that is a TiKV operation and does not involve the local datastore.
