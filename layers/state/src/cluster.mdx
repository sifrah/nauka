The cluster store is Nauka's distributed persistence backend. It stores all shared state that must be visible and consistent across every node in the mesh: virtual machines, VPCs, networks, volumes, users, and other cluster-wide resources.

## How it works

The cluster store wraps TiKV, a distributed key-value database that replicates data across nodes using the Raft consensus protocol. Platform engineers interact with TiKV indirectly through Nauka; there is no need to issue TiKV commands directly.

Data is serialized as JSON and stored in TiKV's raw key-value interface. This means you can inspect stored values with TiKV tooling if needed, and the data will be human-readable.

## Connecting to the cluster

The cluster store connects to TiKV through one or more PD (Placement Driver) endpoints. PD is TiKV's coordination service that tracks which data nodes hold which key ranges. A typical endpoint looks like:

```
http://[fd01::1]:2379
```

These are IPv6 ULA addresses from the mesh's address space, reachable over the WireGuard tunnel. The connection requires the mesh to be up and the WireGuard interface active.

## Key structure

All keys in the cluster store follow the pattern `{namespace}/{key}`. The namespace groups related resources together. For example:

- `vms/vm-001` — a virtual machine record
- `networks/net-prod` — a network definition
- `volumes/vol-data-1` — a storage volume

Nauka constructs these compound keys internally. Platform engineers working with the CLI or API specify resource types and names; the key format is an implementation detail.

## Operations

The cluster store supports the standard set of key-value operations:

**Read and write.** Individual records can be read, written, or deleted by namespace and key. Writing a key that already exists overwrites the previous value. Reading a key that does not exist returns nothing (no error).

**Listing.** Records can be listed by namespace and prefix. For example, listing all keys under `vms/` returns every virtual machine in the cluster. Listing supports up to 10,000 results per query.

**Batch writes.** Multiple records in the same namespace can be written in a single round-trip to TiKV. This reduces latency when creating several related resources at once (for example, provisioning a VM with its network interfaces and volumes).

**Existence checks.** You can check whether a key exists without reading its full value.

## Replication and consistency

TiKV replicates data across multiple nodes using Raft. The default replication factor is three, meaning each piece of data is stored on three separate nodes. If a node goes down, reads and writes continue as long as a majority of replicas are available.

All reads and writes go through the Raft leader for the relevant key range, which provides linearizable consistency. A successful write is guaranteed to be visible to all subsequent reads from any node in the mesh.

## When the cluster store is not available

The cluster store requires TiKV to be running, which in turn requires the mesh to be up. During bootstrap (before the first mesh is created) and during mesh operations (init, join, leave), Nauka uses the local store instead. See the [State Layer overview](/contributing/overview) for details on the two-tier model.

## Failure handling

If TiKV is unreachable or a request fails, the error is surfaced to the caller. There is no automatic retry built into the cluster store layer. Higher-level components decide whether and how to retry based on the operation being performed.

Errors fall into two categories:

- **Database errors** — TiKV is unreachable, a node is down, or a request timed out.
- **Serialization errors** — a stored value cannot be decoded, usually indicating data corruption or a schema change.
