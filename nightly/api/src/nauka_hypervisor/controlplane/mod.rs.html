1//! Control plane — distributed cluster state.
2//!
3//! Manages PD + TiKV as systemd services and exposes [`ClusterDb`] for
4//! distributed state (orgs, projects, envs, vpcs, subnets, vms, ...).
5//! All traffic flows over the encrypted WireGuard mesh.
6//!
7//! # Sub-modules
8//!
9//! - `service`: Low-level systemd management (install, start, stop, reload, status)
10//! - `ops`: High-level orchestration (bootstrap, join, leave)
11//! - `store`: [`ClusterDb`] — thin wrapper around
12//!   [`nauka_state::EmbeddedDb`] on the SurrealDB TiKv backend
13//!
14//! # P2.8 — `ClusterDb` is now a wrapper around `EmbeddedDb<TiKv>`
15//!
16//! P2.8 (sifrah/nauka#212) swapped the underlying transport on
17//! [`ClusterDb`] from the direct `tikv-client::RawClient` to
18//! [`nauka_state::EmbeddedDb::open_tikv`]. The legacy raw-KV surface
19//! (`put`/`get`/`delete`/`list`/`scan_keys`/`exists`/`batch_put`) is
20//! still there — it bridges through a `data`-wrapped JSON record
21//! shape — so every caller (`layers/org`, `layers/network`,
22//! `layers/compute`) still compiles unchanged.
23//!
24//! New code should go through `ClusterDb::embedded()` to reach the
25//! native SurrealDB SDK directly. The legacy wrapper methods exist
26//! only to keep the in-flight cascade (P2.9–P2.14 / sifrah/nauka#213–#218)
27//! compiling. P2.16 (sifrah/nauka#220) deletes the wrapper entirely.
28

53/// Connect to the cluster-side SurrealDB (TiKv backend) using PD
54/// endpoints from local fabric state.
55///
56/// This is the standard way for any layer to get a [`ClusterDb`]
57/// connection. Reads the local hypervisor state to discover PD
58/// endpoints on the mesh, then hands them to
59/// [`ClusterDb::connect_from_addresses`], which in turn drives
60/// [`nauka_state::EmbeddedDb::open_tikv`].
61///
62/// P2.3 (sifrah/nauka#207) replaced the per-call `format!` ladder
63/// with the shared [`crate::fabric::state::FabricState::pd_endpoints`]
64/// and [`nauka_state::pd_endpoints_for`] helpers so every Phase-2
65/// call site lands on the same PD list shape. P2.8 (sifrah/nauka#212)
66/// then routed the final `ClusterDb::connect` through `EmbeddedDb`
67/// so bootstrap (SurrealKV) and cluster (TiKv) state both go through
68/// one wrapper.
69pub async fn connect() -> anyhow::Result<ClusterDb> {
70    let local_db = nauka_state::EmbeddedDb::open_default().await?;
71
72    let state = crate::fabric::state::FabricState::load(&local_db)
73        .await
74        .map_err(|e| anyhow::anyhow!("{e}"))?
75        .ok_or_else(|| {
76            anyhow::anyhow!(
77                "cluster not initialized.\n\n\
78                 Initialize a cluster first with:\n\
79                 \x20 nauka hypervisor init"
80            )
81        })?;
82    // Explicit shutdown so the SurrealKV flock is released before the
83    // caller issues its next `connect()` or local-state read. The
84    // cluster-side `ClusterDb` we return is a separate handle against
85    // TiKv — it does not touch the local `bootstrap.skv` datastore.
86    local_db.shutdown().await?;
87
88    // Self first, peers after. The self-first contract matters for
89    // single-node clusters (nothing in peers) and for always routing
90    // through the cheapest hop when multiple PDs are live.
91    let mesh_addrs = state.pd_endpoints();
92    ClusterDb::connect_from_addresses(&mesh_addrs, PD_CLIENT_PORT)
93        .await
94        .map_err(Into::into)
95}
