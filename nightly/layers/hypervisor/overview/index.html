Hypervisor Layer

Every server running Nauka is a hypervisor. There is no separate controller — each node participates equally in the mesh. The hypervisor layer is what transforms a Linux server into a Nauka node by joining the encrypted WireGuard mesh, bootstrapping or joining the distributed control plane, and configuring S3-backed storage.

Architecture

Section titled “Architecture”

A hypervisor node runs three sub-systems, each managed as systemd services:

graph TD
subgraph Hypervisor Node
  F[Fabric<br/>WireGuard mesh]
  C[Control Plane<br/>PD + TiKV]
  S[Storage<br/>ZeroFS per region]
end
F -->|mesh IPv6| C
F -->|mesh IPv6| S
C -->|region config| S
S -->|S3| B[(Object Storage)]
  • Fabric — WireGuard mesh networking between all nodes. All inter-node traffic flows over encrypted IPv6 tunnels.
  • Control Plane — A TiKV distributed key-value store (with PD for consensus) that holds cluster-wide state. Runs on the mesh.
  • Storage — ZeroFS instances that mount S3 buckets as local block storage, one per region.

All three sub-systems start automatically during init or join and are managed together through the hypervisor CLI.

Lifecycle commands

Section titled “Lifecycle commands”

The hypervisor is accessible as nauka hypervisor, nauka hv, or nauka node — all three are identical aliases.

Initialize a new cluster

Section titled “Initialize a new cluster”

This command:

  1. Generates a mesh identity (WireGuard keypair, ULA /48 network, node /128 address)
  2. Creates the WireGuard interface and systemd service
  3. Installs TiUP and bootstraps PD + TiKV as a single-node cluster
  4. Publishes the region storage configuration to TiKV
  5. Installs ZeroFS and starts the storage service for the given region
  6. Optionally starts the peering listener to accept joins immediately (--peering)

After initialization, the command prints a PIN that other nodes need to join.

Join an existing cluster

Section titled “Join an existing cluster”

The join flow:

  1. Connects to the target node’s peering port (WireGuard listen port + 1)
  2. Authenticates with the PIN, receives the mesh secret and peer list
  3. Configures WireGuard with all known peers
  4. Joins or connects to the TiKV control plane (PD on the first 3 nodes, TiKV-only after that)
  5. Fetches the region’s S3 storage configuration from TiKV and sets up ZeroFS locally
  6. Announces itself to all peers in the mesh

Day-to-day operations

Section titled “Day-to-day operations”

Caution

nauka hv leave is destructive. It deregisters the node’s TiKV store and PD member from the cluster, stops all services, removes systemd units, configuration files, and data directories. This action requires confirmation.

Start and stop order

Section titled “Start and stop order”

When starting services, Nauka brings them up in dependency order: fabric first (WireGuard must be up for mesh communication), then the control plane (PD before TiKV), then storage.

When stopping, the order is reversed: storage services stop first, then the control plane, then the fabric.

Network modes

Section titled “Network modes”

The fabric layer supports three network backends, selected with --mode at init or join time:

ModeDescriptionUse case
wireguard (default)Encrypted WireGuard mesh over IPv6Production deployments over the internet
directPlain IP routing, no tunnelTrusted private networks
mockNo-op backend for testingDevelopment and CI

In direct and mock modes, the control plane and storage sub-systems are not deployed — only the fabric peer tracking runs.

Ports

Section titled “Ports”
PortProtocolServiceBinding
51820UDPWireGuard tunnelPublic IP
51821TCPPeering listener (join requests)Public IP
51822TCPAnnounce listener (peer updates)All interfaces
2379TCPPD client APIMesh IPv6

The WireGuard port is configurable with --port. The peering and announce ports are derived as port + 1 and port + 2. Control plane ports are fixed and bind only to the mesh IPv6 address.
