Peering Server

The peering server is the TLS listener that runs when a Nauka hypervisor is in peering mode. When a new node wants to join the mesh, it connects to this server, which validates the PIN, sends back the mesh credentials and peer list, and then announces the new node to all existing members.

The server listens on WireGuard port + 1 (default 51821). This port must be reachable over TCP from any node that needs to join.

Connection lifecycle

Section titled “Connection lifecycle”
sequenceDiagram
  participant Joiner as Joining Node
  participant Server as Peering Server
  participant DB as State Database
  participant WG as WireGuard
  participant Peers as Existing Peers

  Joiner->>Server: TCP connect
  Server->>Server: Check rate limit (IP not blocked?)
  Server->>Joiner: TLS handshake (self-signed cert)
  Joiner->>Server: JoinRequest (JSON, length-prefixed)
  Server->>Server: Validate PIN
  Server->>Server: Validate peer fields (no injection)
  Server->>DB: Open database, load FabricState
  Server->>Joiner: JoinResponse (secret, prefix, peers)
  Server->>DB: Add new peer, save state
  Server->>WG: Update nauka0.conf + wg syncconf
  Server->>Peers: Broadcast PeerAnnounce (async)

For each incoming connection, the server follows this sequence:

  1. Rate limit check — If the source IP has exceeded the PIN failure threshold, the connection is dropped immediately.
  2. TLS handshake — The server presents a self-signed certificate generated at startup. The certificate is ephemeral and uses the TOFU (Trust on First Use) model.
  3. Read join request — A length-prefixed JSON message is read from the TLS stream.
  4. Validate peer fields — All fields from the request (name, region, zone, public key, endpoint) are checked for control characters and excessive length to prevent injection attacks.
  5. Auto-detect endpoint — If the joiner did not specify an endpoint, the server derives it from the TCP connection’s source IP and the declared WireGuard port.
  6. Validate PIN — The provided PIN is compared against the expected value. On mismatch, a rejection response is sent and the failure is recorded for rate limiting.
  7. Load state — The database is opened and the current fabric state is loaded.
  8. Send response — A join response containing the mesh secret, /48 prefix, mesh ID, full peer list, and the server’s own peer information is sent back to the joiner.
  9. Save state — The new peer is added to the local peer list and state is persisted. State is saved before updating WireGuard, so the database always reflects the true membership even if the WireGuard update fails.
  10. Update WireGuard — The nauka0.conf file is rewritten with the new peer and applied via wg syncconf. If syncconf fails, a service restart is attempted as fallback.
  11. Announce — The new peer’s information is broadcast to all existing peers on their announce ports (port+2) over TLS. This happens asynchronously and does not block the join response.

Rate limiting

Section titled “Rate limiting”

The server implements per-IP rate limiting to protect against PIN brute-force attacks:

ParameterValue
Max failed PIN attempts per IP5
Block duration after threshold10 minutes (600 seconds)
ResetSuccessful join clears the failure counter for that IP

After 5 failed PIN attempts from the same IP address, all subsequent connections from that IP are dropped for 10 minutes. The failure counter resets either when the block period expires or when a successful join occurs from that IP.

Per-request database access

Section titled “Per-request database access”

The database is opened fresh for each join request rather than held open across requests. This is intentional. The embedded database uses a single-writer model, and holding a write lock for the duration of a TCP connection would block any concurrent operation (such as nauka hypervisor status or a health check sweep) that needs to read state.

Opening per-request adds a small amount of latency to joins, but joins are infrequent and not latency-sensitive.

Leave and rejoin handling

Section titled “Leave and rejoin handling”

The server handles the case where a node leaves and then rejoins with the same name but a different WireGuard key:

  • If the exact same public key already exists in the peer list, the join is treated as a duplicate and succeeds without adding a new entry.
  • If the same name exists with a different key (indicating a rejoin after leave), the stale entry is removed and replaced with the new one.
  • The peer list sent in the join response filters out any stale entry for the joining node’s name, so the joiner does not receive its own outdated record.

Firewall requirements

Section titled “Firewall requirements”

The peering port must be open for inbound TCP connections. If you use nftables or iptables:

The WireGuard port (51820 UDP) and announce port (51822 TCP) should also be open. The announce port only needs to be reachable over the mesh network (not from the public internet).
