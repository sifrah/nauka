Peer Announcement

When a node joins or leaves the mesh, every other node needs to know about it. The announce system handles this propagation. Without it, existing nodes would not have WireGuard entries for newly joined peers and could not route traffic to them. Similarly, when a node leaves, the announce system ensures its peers remove the stale WireGuard configuration.

How announcements work

Section titled “How announcements work”

The announce system uses two message types, both sent over TLS connections to each peer’s announce port (wg_port + 2, default 51822).

When a new node joins

Section titled “When a new node joins”
sequenceDiagram
  participant New as New Node
  participant Acceptor as Accepting Node
  participant P1 as Peer 1
  participant P2 as Peer 2

  New->>Acceptor: Join request (peering port)
  Acceptor->>Acceptor: Accept join, update state
  Acceptor->>P1: PeerAnnounce over TLS (announce port)
  Acceptor->>P2: PeerAnnounce over TLS (announce port)
  P1->>P1: Add peer, update WG config
  P2->>P2: Add peer, update WG config

After accepting a join request, the accepting node broadcasts a peer announcement to every existing peer in parallel. Each announcement is a TLS connection to the target peer’s announce port carrying a JSON message that identifies the new node (name, WireGuard public key, endpoint, mesh IPv6 address) and who sent the announcement.

Each receiving peer:

  1. Validates the incoming data against injection attacks
  2. Checks whether the peer is already known (skips if same key exists)
  3. Replaces stale entries if the name matches but the key differs (rejoin scenario)
  4. Adds the new peer to its local peer list
  5. Persists the updated state
  6. Rewrites /etc/wireguard/nauka0.conf and applies changes via wg syncconf

When a node leaves

Section titled “When a node leaves”

Before tearing down its network, a departing node broadcasts a removal message to all known peers. Each receiving peer removes the departing node from its peer list and WireGuard configuration.

The leaving node retries the removal broadcast up to 4 times with 3-second delays between attempts. This handles cases where a peer’s announce listener is temporarily unavailable.

The announce listener

Section titled “The announce listener”

Every node that runs peering mode (nauka hypervisor peering) starts an announce listener on wg_port + 2 (default 51822). This listener:

  • Binds to [::] (all interfaces, dual-stack IPv4/IPv6)
  • Accepts TLS connections using the same TOFU model as the peering server
  • Opens a fresh database connection for each incoming message (avoids lock contention)
  • Processes both peer announcements and peer removals
  • Runs indefinitely until the peering process is stopped

The announce listener also has a dedicated systemd service (nauka-announce) that can run persistently:

ItemValue
Service namenauka-announce
Unit file/etc/systemd/system/nauka-announce.service
Depends onnauka-wg.service
Restart policyOn failure, 5-second delay

This service is managed through the nauka hypervisor commands and does not need manual intervention.

Port allocation

Section titled “Port allocation”
PurposeDefault PortProtocol
WireGuard data plane51820UDP
Peering listener (join requests)51821TCP/TLS
Announce listener51822TCP/TLS

The announce port is always wg_port + 2. When broadcasting, the sending node connects to [mesh_ipv6]:51822 on each peer — announcements travel over the mesh network, not over the public internet.

Best-effort delivery and convergence

Section titled “Best-effort delivery and convergence”

Individual announcements are fire-and-forget. If a peer is unreachable or its announce listener is not yet running (common during concurrent joins), the failure is logged and the broadcast continues to remaining peers. A failed announcement does not fail the join.

The system achieves reliable convergence through periodic reconciliation rather than guaranteed delivery. Every 30 seconds, the peering node re-reads its peer list and re-announces any peers added within the last 60 seconds. This sweep is idempotent — peers that already know about the announced node skip the update silently.

This means the mesh converges to full connectivity within approximately 60 seconds of the last join, even in worst-case scenarios with many concurrent joins, timing issues, or transient network problems.

Idempotency

Section titled “Idempotency”

The announce handler checks whether an incoming peer is already known before adding it:

  • Same public key already known — The announcement is silently skipped. This handles duplicate announcements from multiple sources or reconciliation sweeps.
  • Same name, different key — The old entry is replaced with the new one. This handles the leave-and-rejoin scenario where a node generates a new keypair.

This design means it is always safe to send redundant announcements. The reconciliation loop exploits this property to guarantee convergence without tracking which peers have received which announcements.

Security

Section titled “Security”

All announce connections use TLS with the same TOFU (Trust on First Use) model as the peering protocol. The TLS layer prevents eavesdropping and injection by attackers on the local network. The WireGuard mesh itself provides an additional layer of encryption at the transport level, since announce traffic travels over mesh IPv6 addresses.

All incoming peer data is validated against injection attacks (control characters, excessive length) before being used in WireGuard configuration files.
