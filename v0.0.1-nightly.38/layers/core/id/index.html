Typed Resource IDs

Every resource in Nauka has a unique identifier. These aren’t plain UUIDs — they are typed, sortable, and human-readable at a glance.

For example:

The prefix tells you what kind of resource it is. The 26-character suffix is a ULID encoded in Crockford Base32 — 48 bits of millisecond timestamp followed by 80 bits of randomness.

Why ULIDs instead of UUIDs?

Section titled “Why ULIDs instead of UUIDs?”

UUIDs are random. Two UUIDs give you no information about which was created first. ULIDs embed a timestamp, so they sort chronologically. A list of resource IDs sorted lexicographically is also sorted by creation time.

Parsing and validation

Section titled “Parsing and validation”

IDs are validated on parse. An ID string that doesn’t match {prefix}-{26-chars} or contains invalid Crockford Base32 characters is rejected immediately with a descriptive error.

Names vs. IDs

Section titled “Names vs. IDs”

Resources can have both a name and an ID. Names are for humans — they’re mutable, descriptive, and subject to naming rules (see validate). IDs are for machines — they’re immutable, globally unique, and used in all API calls and state references.

When you reference a resource in the CLI, you can often use the name. Internally, everything uses the ID. The distinction matters when scripting: if you rename a VPC, scripts referencing it by name break. Scripts referencing it by ID do not.
