Resource Registry

The resource registry is the central catalog of everything Nauka can manage. At startup, each layer registers its resource definitions along with the handler functions that implement them. The CLI and API then draw from this registry to build their interfaces.

How registration works

Section titled “How registration works”
flowchart LR
  HL[Hypervisor Layer] -->|registers| R[Resource Registry]
  SL[State Layer] -->|registers| R
  CL[Core Layer] -->|registers| R
  R --> CLI[CLI Command Tree]
  R --> API[REST API Router]

Each layer is responsible for registering its own resources during initialization. A registration consists of two things:

  1. A resource definition — the complete description of the resource (identity, scope, schema, operations, presentation).
  2. A handler function — the async function that executes operations for this resource.

The handler receives an operation request and returns a response. It is responsible for interpreting the operation name, extracting field values, performing the actual work (reading/writing state, managing services, etc.), and returning the appropriate response type.

What the registry provides

Section titled “What the registry provides”

Once all layers have registered their resources, the registry enables:

  • CLI generation: The registry is iterated to build the complete command tree. Each registered resource becomes a top-level subcommand.
  • API routing: The registry is iterated to build the REST API router. Each resource’s operations become API endpoints.
  • Resource lookup: Given a CLI command name or alias, the registry finds the matching resource definition and handler.

Startup sequence

Section titled “Startup sequence”

The typical startup sequence for a Nauka binary:

  1. Create a new, empty registry.
  2. Initialize each layer, which registers its resources with the registry.
  3. Build the CLI command tree from the registry.
  4. Build the API router from the registry (if running in server mode).
  5. Run the requested command or start the server.

The binary itself has no hardcoded knowledge of what resources exist. It simply iterates whatever has been registered. This means adding a new resource to a layer requires no changes to the binary — the resource is automatically discovered.

Looking up resources

Section titled “Looking up resources”

The registry supports looking up resources by their CLI name or any of their aliases. For example, if hypervisor is registered with the alias hv, both nauka hypervisor list and nauka hv list resolve to the same resource definition and handler.

Request and response types

Section titled “Request and response types”

Handlers communicate with the dispatch layer through two types:

Operation request — contains the operation name, the target resource name (if applicable), scope values (parent identifiers), and all parsed field values. By the time the handler receives this, all arguments have been parsed, type-checked, and constraint-validated.

Operation response — the handler returns one of:

ResponseUsed for
Single resource (JSON)get, create, update results
Resource list (JSON array)list results
Text messageConfirmations, status messages
NoneSilent operations

The dispatch layer handles rendering the response as a table, detail view, plain text, or JSON depending on the operation type and the --json flag.

Self-registering architecture

Section titled “Self-registering architecture”

This architecture means that layers are fully self-contained. Each layer declares what it manages and how to handle operations on those resources. The binary is just a thin composition layer that creates a registry, lets each layer register, and then runs the requested command.

To see how a specific resource is defined and registered, look at the handlers.rs file in the relevant layer (e.g., layers/hypervisor/src/handlers.rs for the hypervisor resource).
