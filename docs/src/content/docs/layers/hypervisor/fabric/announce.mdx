When a new node joins the mesh, the accepting hypervisor needs to tell every other existing peer about it. When a node leaves, it needs to tell peers to remove it. The announce system handles both directions, plus a persistent listener on every node that receives these updates.

## Message types

The announce protocol uses a tagged JSON envelope (`AnnounceMessage`) with two variants:

### PeerAnnounce

Broadcast when a new node joins. Each receiving peer adds the node to its WireGuard config and peer list.

```json
{
  "type": "announce",
  "peer": {
    "name": "web-03",
    "region": "eu",
    "zone": "fsn1",
    "wg_public_key": "...",
    "wg_port": 51820,
    "endpoint": "203.0.113.30:51820",
    "mesh_ipv6": "fd01::3"
  },
  "announced_by": "web-01"
}
```

### PeerRemove

Broadcast when a node leaves. Each receiving peer removes the node from its WireGuard config and peer list.

```json
{
  "type": "remove",
  "name": "web-03",
  "wg_public_key": "..."
}
```

## Flow

### After a join

1. The accepting node (peering server) calls `broadcast_new_peer()` over TLS to each existing peer's announce port
2. Each receiving peer adds the new node, persists state, and updates WireGuard config
3. If any announcements fail (peer's listener not ready yet), the peering server retries after 30 seconds with a **full mesh reconciliation** — it re-reads the complete peer list and announces every peer to every other peer
4. The joining node also **self-announces** to all peers it knows, covering the case where the initial broadcast arrived before a peer's listener was ready

### After a leave

1. The leaving node calls `broadcast_peer_remove()` over TLS to all known peers
2. Each receiving peer removes the node from its peer list and WireGuard config
3. The leaving node then tears down its own network and deletes state

## Persistent announce listener (`nauka-announce`)

Every node runs a systemd service called `nauka-announce` that listens on `wg_port + 2` (default 51822) for incoming announcements. This service is:

- **Installed** automatically after `init` or `join`
- **Started/stopped** with `nauka hv start` and `nauka hv stop`
- **Removed** during `nauka hv leave`
- **Not installed** during `init --peering` (the peering listener runs its own inline announce listener to avoid port conflicts; the service is installed after peering exits)

The service runs `nauka hypervisor announce-listen --port <port>`, which is a hidden CLI command that blocks forever, processing incoming announce messages.

Without this persistent listener, nodes that join concurrently could miss peer announcements because no one would be listening at the time the announcement was sent.

## Port allocation

| Purpose | Port |
|---|---|
| WireGuard data plane | `wg_port` (default 51820) |
| Peering listener (join requests) | `wg_port + 1` (51821) |
| Announce listener | `wg_port + 2` (51822) |

All listeners bind to `[::]` (dual-stack IPv4+IPv6) since mesh traffic uses ULA IPv6 addresses.

## Leave/rejoin handling

When a node leaves and rejoins with a new keypair:

- The **peering server** filters out stale entries for the rejoining node's name before sending the peer list
- The **announce handler** replaces a stale entry (same name, different key) instead of silently skipping
- Other peers receive the `PeerRemove` on leave and the `PeerAnnounce` on rejoin — no stale WireGuard entries accumulate

## Periodic mesh reconciliation

The peering node (the one running `--peering` or `nauka hv peering`) runs a background reconciliation loop every 30 seconds. On each cycle it:

1. Re-reads the complete peer list from the database
2. For each peer, announces it to all other peers
3. Receivers skip peers they already know (idempotent)

This guarantees **eventual full mesh convergence** — even if initial announcements fail due to concurrent joins, timing issues, or transient network problems, the periodic sweep will fix any gaps within ~60 seconds (2 cycles).

## Best-effort delivery

Individual announcements are fire-and-forget per peer. If a peer is unreachable, the failure is logged and the broadcast continues. The periodic reconciliation loop provides a safety net — there is no need for manual intervention after failed announcements.

## Security

All announce connections use TLS with the same TOFU (Trust on First Use) model as the peering protocol — self-signed certificates, no verification. The mesh network itself (WireGuard) provides encryption at the transport layer; TLS prevents injection of fake announcements by attackers on the local network.
