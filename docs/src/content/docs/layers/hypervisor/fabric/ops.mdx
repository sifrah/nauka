The `ops` module is where the work actually happens. Every high-level `nauka hv` command ultimately calls a function here. Ops orchestrates the other fabric sub-modules â€” identity, WireGuard, systemd, peering, state â€” into coherent multi-step workflows.

## InitConfig

`init` accepts an `InitConfig` struct and `join` accepts a `JoinConfig` struct that consolidate all parameters:

**InitConfig:**

| Field | Flag | Description |
|---|---|---|
| `node_name` | `--name` | Node name (defaults to hostname if omitted) |
| `region` | `--region` | Region identifier (e.g., `us-east`) |
| `zone` | `--zone` | Availability zone (e.g., `us-east-1a`) |
| `port` | `--port` | Listen port (also uses port+1 for peering, port+2 for announce) |
| `network_mode` | `--mode` | Network backend: `wireguard` (default), `direct`, or `mock` |
| `fabric_interface` | `--interface` | Physical network interface (e.g., `eth1`). Auto-detected if omitted |
| `endpoint` | `--endpoint` | This node's public endpoint (`host:port`) advertised to peers |

**JoinConfig:**

| Field | Flag | Description |
|---|---|---|
| `target` | `--target` | IP or IP:port of an existing node (required) |
| `node_name` | `--name` | Node name (defaults to hostname if omitted) |
| `region` | `--region` | Region identifier |
| `zone` | `--zone` | Availability zone |
| `port` | `--port` | Listen port |
| `pin` | `--pin` | PIN for auto-accept |
| `network_mode` | `--mode` | Network backend |

The `--mode` flag selects the [network backend](/nauka/layers/hypervisor/fabric/backend). All nodes in a cluster should use the same mode.

The `--endpoint` flag is optional. If omitted, Nauka detects the public IP automatically. Set it explicitly when running behind NAT or when you want to control which address peers connect to.

## init

`init` bootstraps the first node in a new mesh.

**What it does:**

1. Generates a `MeshIdentity` (UUID + `/48` prefix) via `create_mesh()`
2. Generates a `HypervisorIdentity` (WG keypair, assigns mesh IPv6) via `create_hypervisor()`
3. Selects the network backend from `InitConfig.mode` via `create_backend()`
4. Installs the backend service via the [service module](/nauka/layers/hypervisor/fabric/service)
5. Starts the service
6. Persists `FabricState` (including `network_mode`) to `~/.nauka/fabric.redb`
7. Installs the `nauka-announce` systemd service (persistent announce listener on `wg_port + 2`)
8. Derives a short PIN from the mesh secret for use during peer joins

**Usage:**

```sh
nauka hv init --name web-01 --region us-east --zone us-east-1a
nauka hv init --name web-01 --region us-east --zone us-east-1a --mode direct
nauka hv init --name web-01 --region us-east --zone us-east-1a --endpoint 203.0.113.10:51820
# Mesh secret: syf_sk_AbC123...
# PIN: 7842
```

If any step fails, init cleans up (stops and uninstalls the service, removes state) so you can safely retry.

## join

`join` connects a new node to an existing mesh.

**What it does:**

1. Ensures the network backend is installed via `backend.ensure_installed()`
2. Connects to the target's peering port (WG port + 1) via TLS+TCP
3. Sends a `JoinRequest` with name, region, zone, WG pubkey, and PIN
4. Receives a `JoinResponse` containing the mesh secret, the `/48` prefix, and the full peer list
5. Creates a `HypervisorIdentity` for this node using the received mesh data
6. Sets up the network via `backend.setup()` with all received peers pre-configured
7. Persists `FabricState` (including `network_mode`)
8. Installs the `nauka-announce` systemd service (persistent announce listener)
9. Self-announces to all known peers after a short delay, ensuring peers that were still joining when the initial broadcast happened can discover this node

**Usage:**

```sh
nauka hv join --target 203.0.113.10 --pin G7CCZX --name web-02 --region us-east --zone us-east-1a
```

The `--name` flag is optional (defaults to hostname). The target must be a running Nauka hypervisor with peering active. You need the PIN displayed when the mesh was initialized.

### Concurrent joins

Multiple nodes can join the same mesh simultaneously. The peering server processes joins sequentially (one at a time), and each joiner receives the full peer list known at that moment. Three mechanisms ensure all nodes discover each other even when joins overlap:

1. **Persistent announce listener** â€” each node runs `nauka-announce.service` (installed during join), which listens on `wg_port + 2` for incoming peer announcements indefinitely. Without this, nodes that join concurrently would miss each other's broadcasts.
2. **Self-announce** â€” after joining, each node announces itself to all known peers. This covers the case where the peering server's broadcast arrived before a peer's announce listener was running.
3. **Periodic mesh reconciliation** â€” the peering node (the one running `--peering` or `nauka hv peering`) runs a background reconciliation loop every 30 seconds. It re-reads the complete peer list from the database and announces every peer to every other peer. This is idempotent â€” peers that already know each other skip the announcement. This guarantees **eventual full mesh convergence** even in the worst case of many concurrent joins.

Convergence time: the mesh is fully connected within ~60 seconds (2 reconciliation cycles) of the last join, regardless of join ordering or timing.

## listen_for_peers

`listen_for_peers` starts the peering mode for this node. It spawns three background tasks and then runs until cancelled:

1. **Peering listener** â€” TCP server on `wg_port + 1` (default 51821) that accepts incoming `join` requests. For each request it opens a fresh DB connection, validates the PIN, sends the mesh secret and peer list to the joiner, and calls `broadcast_new_peer()` to notify existing peers.
2. **Health check loop** â€” polls WireGuard handshake timestamps every 30 seconds and updates peer status (`Active` / `Unreachable`). See [Health Checks](/layers/hypervisor/fabric/health).
3. **Announce listener** â€” TCP server on `wg_port + 2` (default 51822) that receives `PeerAnnounce` messages from other nodes and adds newly joined peers to this node's WireGuard config. See [Peer Announcement](/layers/hypervisor/fabric/announce).

**Port allocation summary:**

| Purpose | Port |
|---|---|
| WireGuard data plane | 51820 (`wg_port`) |
| Peering listener | 51821 (`wg_port + 1`) |
| Announce listener | 51822 (`wg_port + 2`) |

All three tasks run concurrently. The per-request database open on the peering listener is intentional â€” it prevents one slow join from blocking other operations.

## status

`status` loads `FabricState`, queries the WireGuard interface, and checks the systemd service state. It returns:

- Mesh ID and prefix
- This node's name, region, zone, mesh IPv6
- Service status (running / stopped)
- WireGuard listen port, peer count, RX/TX bytes
- List of peers with their current status

```sh
nauka hv status
```

## start

`start` starts all hypervisor services: fabric (network), control plane (PD + TiKV), and storage (ZeroFS regions). If a service is already running, it is skipped. Warnings are printed for any service that fails to start. State must exist (i.e. the node must have been initialized) for start to work.

```sh
nauka hv start
```

## stop

`stop` stops all hypervisor services in reverse order: storage first, then control plane, then fabric. State is preserved â€” you can `start` again to reconnect.

```sh
nauka hv stop
```

## leave

`leave` permanently removes this node from the mesh.

**What it does:**

1. Stops and removes all storage services (ZeroFS instances)
2. Deregisters this node's TiKV store and PD member, then uninstalls control plane services
3. Uninstalls the `nauka-announce` service
4. Broadcasts a `PeerRemove` message to all known peers over TLS â€” each peer removes this node from its WireGuard config and peer list
5. Tears down the fabric network via the backend
6. Deletes `FabricState` from the database

After `leave`, the node is clean. Other nodes receive the `PeerRemove` and immediately clean up â€” no stale entries remain. To rejoin, run `nauka hv join` again.

```sh
nauka hv leave
```
