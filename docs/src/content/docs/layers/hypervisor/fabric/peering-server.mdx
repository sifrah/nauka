The peering server is the TCP listener that runs on every active Nauka hypervisor. When another node wants to join the mesh, it connects here. The server validates the PIN, sends back the mesh secret and peer list, and then announces the new node to everyone else.

The server listens on **WireGuard port + 1**. If your WireGuard port is 51820, the peering server is on 51821. Make sure this port is reachable from nodes that need to join.

## Connection lifecycle

For each incoming connection, the server follows this sequence:

1. **Accept TCP connection** and perform **TLS handshake** (self-signed cert, TOFU model)
2. **Read JoinRequest** (length-prefixed JSON over TLS)
3. **Check rate limit** — IPs with too many failed PIN attempts are blocked for 10 minutes
4. **Validate PIN** — if wrong, send rejection and record failure
5. **Validate peer fields** — reject names, keys, or endpoints containing control characters
6. **Open the database** to read current `FabricState`
7. **Build JoinResponse** with the mesh secret, prefix, and current peer list
8. **Send JoinResponse** to the joiner
9. **Update WireGuard config** with the new peer (before persisting state — no split-brain)
10. **Persist the new peer** to local `FabricState`
11. **Announce the new peer** to all existing peers via TLS `PeerAnnounce` on port wg_port+2
12. **Close the connection**

## Per-request database open

The database is opened fresh for each join request rather than held open across requests. This is a deliberate design choice. redb uses a single-writer model, and holding a write lock for the duration of a TCP connection would block any other operation (like `nauka hv status`) that needs to read state while a slow join is in progress.

Opening per-request adds a small amount of latency to joins, but joins are rare and not latency-sensitive.

## Announcing new peers

After a join completes, the server connects to each existing peer's announce port (wg_port+2) over TLS and sends a `PeerAnnounce` message. Each existing peer then:

- Adds the new node to its own `PeerList`
- Updates the WireGuard config and applies it via `wg syncconf`
- Persists state to disk

If any announcement fails (e.g. the target peer's announce listener isn't ready due to a concurrent join), the peering server's **periodic reconciliation loop** will fix it. Every 30 seconds, the peering node re-reads the complete peer list and announces every peer to every other peer. This is idempotent and guarantees full mesh convergence within ~60 seconds.

This means a new node is fully connected to the whole mesh within seconds of a successful join (or at most ~60 seconds with concurrent joins) — no manual configuration on other nodes is needed.

## Firewall requirements

The peering port must be open on the accepting node. If you use nftables or iptables:

```sh
# Allow peering connections (WG port + 1) and announce (WG port + 2)
nft add rule inet filter input tcp dport 51821 accept
nft add rule inet filter input tcp dport 51822 accept
```

The WireGuard port itself (51820 by default) also needs to be open for UDP traffic. The announce port (51822) must be reachable over the mesh for peer announcements to propagate.

## Logging

The peering server logs each connection attempt, including the source IP and whether the PIN was accepted or rejected. This gives you an audit trail of who has joined the mesh and when.
