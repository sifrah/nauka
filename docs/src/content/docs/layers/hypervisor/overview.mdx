In Nauka, every server is a hypervisor. There is no separate "controller" host — each node in your cluster is both a participant and a peer. The hypervisor layer is what turns a plain Linux server into a Nauka node: it joins the WireGuard mesh, tracks peers, manages the tunnel service, and exposes a CLI for day-to-day operations.

## What the hypervisor does

When you initialize a server with Nauka, the hypervisor layer:

1. Generates a WireGuard keypair and a mesh identity for the node
2. Creates the WireGuard interface (`nauka0`) and a systemd service to manage it
3. Persists the node's identity and peer list to the [state layer](/nauka/layers/state/overview)
4. Listens for incoming join requests so other nodes can peer with it

After initialization, the node is a full member of the mesh. Every other node that joins will establish a direct encrypted tunnel to it.

## Aliases

The hypervisor is exposed as both `hv` and `node` in the CLI. They are identical — use whichever feels natural.

```sh
nauka hv status
nauka node status
```

## Commands

### init

Create a new mesh and initialize this server as the first hypervisor.

```sh
nauka hv init --name web-01 --region us-east --zone us-east-1a
```

This generates the mesh secret (a `syf_sk_...` token), creates the WireGuard service, and starts it. Save the mesh secret — other nodes need it to join.

### join

Add this server to an existing mesh by connecting to a peer that is already a member.

```sh
nauka hv join --target 203.0.113.10 --name web-02 --region us-east --zone us-east-1a
```

The join command connects to the target's peering port, authenticates with the mesh PIN, receives the mesh secret and existing peer list, then brings up the WireGuard service with all known peers configured.

### status

Show the current state of this node: mesh identity, WireGuard service health, and connected peer count.

```sh
nauka hv status
```

### start / stop

Start or stop the WireGuard service on this node. Both commands are idempotent — running start on an already-running node is safe.

```sh
nauka hv start
nauka hv stop
```

### leave

Gracefully remove this node from the mesh. Stops and uninstalls the WireGuard service, removes the interface, and deletes local state.

```sh
nauka hv leave
```

### peering

Manage inbound join requests. By default, a running hypervisor listens for peers automatically. Use this command to inspect or control that listener.

```sh
nauka hv peering
```

### list / get

List all known peers or inspect a specific one.

```sh
nauka hv list
nauka hv get web-01
```

### doctor

Run a full health diagnostic across all sub-systems:

```sh
nauka hv doctor
```

Checks fabric, controlplane, storage, and system resources. Prints colored pass/warn/fail output and exits non-zero if any check fails. See [Doctor](/nauka/layers/hypervisor/doctor) for details.

## Network backend modes

The fabric layer supports three network backends selected with `--mode` at init or join time:

| Mode | Description |
|---|---|
| `wireguard` (default) | Encrypted WireGuard mesh — use this for any public internet deployment |
| `direct` | Plain IP routing, no tunnel — for trusted private networks |
| `mock` | No-op backend for testing |

```sh
nauka hv init --name web-01 --mode wireguard
nauka hv join --target 203.0.113.10 --name web-02 --mode direct
```

Two additional flags control the WireGuard interface:

- `--interface` — override the interface name (default: `nauka0`)
- `--endpoint` — explicitly set this node's public endpoint (`host:port`) advertised to peers

## Sub-modules

The hypervisor layer is built on the [fabric](/nauka/layers/hypervisor/fabric/mesh) sub-module, which handles the lower-level details:

- [mesh](/nauka/layers/hypervisor/fabric/mesh) — mesh and node identity
- [peer](/nauka/layers/hypervisor/fabric/peer) — peer tracking and status
- [ops](/nauka/layers/hypervisor/fabric/ops) — high-level orchestration (init, join, leave, etc.)
- [wg](/nauka/layers/hypervisor/fabric/wg) — WireGuard interface management
- [service](/nauka/layers/hypervisor/fabric/service) — systemd service lifecycle
- [peering](/nauka/layers/hypervisor/fabric/peering) — join protocol types
- [backend](/nauka/layers/hypervisor/fabric/backend) — network backend trait and mode selection
- [tls](/nauka/layers/hypervisor/fabric/tls) — TLS for the peering protocol

### Controlplane

The [controlplane](/nauka/layers/hypervisor/controlplane/overview) sub-module manages the distributed TiKV key-value cluster:

- [overview](/nauka/layers/hypervisor/controlplane/overview) — TiKV + PD architecture, cluster topology
- [service](/nauka/layers/hypervisor/controlplane/service) — install, configure, and manage PD + TiKV services
- [ops](/nauka/layers/hypervisor/controlplane/ops) — bootstrap, join, and leave the control plane
- [store](/nauka/layers/hypervisor/controlplane/store) — `ClusterDb` async client for distributed state

### Storage

The [storage](/nauka/layers/hypervisor/storage/overview) sub-module manages ZeroFS S3-backed block storage:

- [overview](/nauka/layers/hypervisor/storage/overview) — ZeroFS, S3 regions, encryption, NBD
- [region](/nauka/layers/hypervisor/storage/region) — `RegionStorage` config and `RegionRegistry`
- [service](/nauka/layers/hypervisor/storage/service) — ZeroFS installation and per-region systemd units
