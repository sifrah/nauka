Every request that reaches a Nauka API server passes through a middleware stack before hitting the resource handler. The middleware is applied in a fixed order and handles cross-cutting concerns that every endpoint needs.

## Rate limiting

The rate limiter enforces per-IP request limits to prevent abuse. By default:

- **Sustained rate:** 100 requests per second per IP
- **Burst allowance:** 500 requests (token bucket)

When the limit is exceeded, the server returns `429 Too Many Requests` with a `Retry-After` header indicating how many seconds to wait:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 2
Content-Type: application/json

{
  "code": "rate_limited",
  "message": "Too many requests. Slow down.",
  "suggestion": "Wait 2 seconds before retrying."
}
```

Rate limits are configurable per deployment:

```rust
ApiServerConfig {
    rate_limit: RateLimitConfig {
        requests_per_second: 50,
        burst: 200,
    },
    ..
}
```

## X-Request-Id

Every request gets a unique request ID, either from an incoming `X-Request-Id` header (which operators can set to correlate with their own tracing) or generated by the server as a new ULID.

The request ID is:
- Attached to all log lines for the duration of the request
- Returned in the `X-Request-Id` response header
- Included in error responses for support tracing

```http
HTTP/1.1 200 OK
X-Request-Id: req-01HWKZ3P8B0E7D2N5FVYMR4CXJ
```

If you report a bug or unexpected error, including the `X-Request-Id` value lets the server operator trace exactly what happened.

## X-Nauka-Version

Every response includes a `X-Nauka-Version` header with the server's version string:

```http
HTTP/1.1 200 OK
X-Nauka-Version: 0.3.1
```

This lets clients detect version mismatches without a separate `/version` endpoint call. The CLI uses this header to warn when the server and client versions are significantly different.

## Pagination

List endpoints accept `limit` and `offset` query parameters. The pagination middleware normalizes these before they reach the handler:

| Parameter | Default | Maximum |
|-----------|---------|---------|
| `limit` | 20 | 200 |
| `offset` | 0 | — |

If `limit` exceeds the maximum, it is clamped to 200. If `offset` is negative, the request is rejected with `400 Bad Request`.

Paginated responses include a `X-Total-Count` header with the total number of items (before pagination), so clients can determine if there are more pages:

```http
HTTP/1.1 200 OK
X-Total-Count: 847
X-Nauka-Version: 0.3.1
Content-Type: application/json

[...]
```

A next-page URL is also included in the response body when there are more results:

```json
{
  "items": [...],
  "total": 847,
  "limit": 20,
  "offset": 40,
  "next": "/vms?limit=20&offset=60"
}
```

## Applying middleware to custom routes

If you add routes outside of the generated resource router, you can apply the standard middleware stack to them:

```rust
use nauka_core::api::middleware::standard_middleware;

let custom_route = axum::Router::new()
    .route("/my-endpoint", get(my_handler))
    .layer(standard_middleware(&config));
```

This ensures custom routes get the same request ID, version header, and rate limiting behavior as generated routes.
